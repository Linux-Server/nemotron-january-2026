# Spike 0.1b — Native launch-overlap microbench (spec)

**Role:** the conjunct-2 *proof* and the conjunct-1 *upside measurement*. Post-Python validation
(`proj-2026-05-24-0859/validation.md`) confirmed the L40S wall is the **single-thread asyncio intake**
(`vad_stop_recv_to_process`) with the GPU **40–65% idle** and compute fine at all loads — and that **adding Python
processes doesn't help (K=3 ≡ K=4 ≈ 16–20/box; MPS/BW contention cancels added procs).** 0.1b answers the one question
that gates the whole project: **does one native process with multi-thread (no-GIL) intake reclaim that idle GPU, and to
what sustainable streams/box before the next bottleneck (MPS/BW contention / finalize-compute) binds?**

## The question (binary + a number)
- **Binary (conjunct 2):** does M-thread no-GIL intake raise the sustainable L40S density above the Python single-thread
  ceiling (~16–20/box)?
- **Number (conjunct 1 upside):** the native density ceiling on L40S = streams/box at the SLO when intake is no longer
  the wall → feeds the 0.0 threshold.

## What to build (minimal — reuses the captured graph, NO decode equivalence)
A small C++ (and/or Rust, to also exercise the tch-rs path) harness that mimics the steady+finalize GPU workload
**without** reimplementing NeMo:
1. **Load the captured steady encoder CUDA graph** (the same bucket the Python server captures — `cudagraph_encoder.py`;
   export it once for the harness) for replay. If graph export isn't ready, use a representative kernel sequence of the
   same shape/cost as a stand-in (document the approximation).
2. **Mock decode** of the right *cost*, not the right *math*: a GPU op of ~the measured `model_wall` per chunk **plus a
   host-side stand-in** for the decode's CPU/`.item()` cost. **Calibrate against the Python measurements** (model_wall
   ≤35 ms; the per-frame decode host time) — this is the load-bearing calibration (see Faithfulness).
3. **N simulated streams**, each emitting a chunk every **160 ms** (steady cadence) + a periodic finalize, fed through:
4. **Intake/dispatch on M real OS threads (no GIL)** — the variable. A/B: **M=1** (mimics the Python single-intake
   baseline) vs **M=cores** (the native thesis).

## A/B + ablation axes
- intake threads **M = 1 vs N_cores** (the core A/B).
- topology: **single-process/single-context vs MPS vs multi-process** (the K=3≡K=4 result predicts single-context
  multi-thread should win — confirm).
- per-lane stream + CUDA-event dependency vs `stream.synchronize` (cheap to include).
- platforms: **5090 (local, fast iterate) + L40S (deploy-relevant).** **NOT L4** (BW-bound; native won't help — confirmed).

## Measure
- **Sustainable streams/box at the SLO** (same criterion as the Python sweeps: vad_stop→final p95 in budget AND
  `vad_stop_recv_to_process` bounded) — for M=1 vs M=cores.
- **GPU utilization** at that point (does it rise from ~40–65%-idle toward saturation?).
- **The next bottleneck** once intake is removed: MPS/BW contention, finalize-compute, or host. Report which binds.

## Pre-registered Go/No-go (freeze before running)
- **Go (conjunct 2 holds + upside real):** M-thread intake raises sustainable L40S density to **≥1.5× → ≥~28/box**
  (SET 2026-05-24) the ~16–20/box baseline, at the SLO, with GPU util rising toward saturation. Record the ceiling
  number for 0.0. **This is the project's gating number** (conjunct 1 is otherwise resolved: L40S fleet + strategic bet).
- **No-go:** the knee doesn't move materially (intake wasn't the sole wall, or MPS/BW contention binds almost
  immediately) → native density upside is small → **STOP** (per the 0.4 tree).

## Faithfulness caveats (state in the result)
- 0.1b measures the **intake-parallelism ceiling assuming the decode costs what we modeled.** If the mock decode is too
  cheap, the ceiling is overestimated. → calibrate the mock against real Python `model_wall` + decode host time; report
  sensitivity (rerun with decode cost ±50%).
- 0.1b proves the **upside EXISTS and bounds it** — it does **NOT** prove B1 is byte-exact or that the real decode hits
  that cost; that's **0.6a** (decode equivalence). 0.1b is the cheap gate *before* funding 0.6a.

## Effort / sequencing
~1–2 eng-wk (reuses the captured graph, no decode equivalence, no full runtime). Run **after** the libtorch version pin
(needed to capture/export the graph) but **before** any Wave-2 native port. It is the cheapest experiment that converts
"~28–40/box TBM" into a real number and is the gate on funding 0.6a/0.2/0.8.
