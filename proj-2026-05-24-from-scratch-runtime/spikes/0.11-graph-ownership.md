# Spike 0.11 — CUDA-graph ownership model (analysis COMPLETE; one number TBM on GPU)

**Goal (PLAN §6 / 0.11):** decide how a multi-threaded, shared-weight native runtime owns CUDA-graph capture/replay,
because the **40–48/box density number depends on it**. The existing Python design sidesteps the hard part with
single-worker executors + per-lane streams; a native multi-lane runtime cannot.

## Constraints from source (verified)
- **Capture is single-threaded today.** The default encoder graph is captured on a dedicated single-worker executor
  (`server.py:1883-1912`), and the code records the capture thread id and *logs when replay happens on a different
  thread* (`server.py:2969`, `:3002`) — i.e. capture/replay thread affinity is already treated as load-bearing.
- **NeMo's own label-looping graph compiler** uses a dedicated capture stream + `capture_error_mode="thread_local"`
  (`rnnt_label_looping.py:839-890`) — confirming capture is not a free-threaded operation.
- **Graphs own static buffers.** Each bucket allocates static input/output tensors (`cudagraph_encoder.py:193-210`),
  replay copies inputs in and returns the **static output pool** (`:272-281`), and the module doc requires callers to
  **clone outputs before the next replay** (`:15-17`). Replay is exact-shape, fail-closed (`:578-599`).
- **Exact-B only.** The manager captures one graph per exact B and never pads (`cudagraph_encoder.py:6-8`); B outside
  the captured range → eager (`server.py:2080-2084`).
- **Memory is already the cap.** L40S is K=3 because K=4 OOMs with model + lane + finalize graph pools
  (`deploy/launch_multiproc.sh:6-9`).

## The three-way tension (PLAN §1 / round 3)
"Shared read-only **weights**" does NOT mean shared **graph buffers**: with per-lane dispatcher threads, each lane that
replays a graph needs its own static-buffer set per B → graph-pool memory is **× lanes**, which is the very K×
duplication the density story claims to remove. Capture must be serialized. Continuous batching (B varies 1..K) keyed to
exact-B graphs means K×lanes resident graphs.

## Options + decision criteria

| Option | Memory | Throughput | Complexity | When to choose |
|---|---|---|---|---|
| **A. Per-lane graph + static-buffer pools** | **× lanes** (worst; may re-hit the K-cap) | best (true parallel replay) | med | only if measured resident pool at target K×lanes FITS L4/L40S (the TBM number below) |
| **B. One shared graph set + serialized replay mutex** | × 1 (best) | reduced (replay serializes → re-introduces a serial lane; measure lost overlap) | med | if memory is tight AND the measured lost-overlap is small vs the tail win |
| **C. No steady graphs (rely on no-GIL lower per-launch overhead)** | × 1 | depends on launch-bound headroom | low | if 0.1 shows no-GIL dispatch alone lifts the knee enough; simplest |

**Required rules for whichever option (capture/allocator/lifetime):**
1. **Capture serialized**, on a known stream, with capture-thread affinity recorded (mirror `server.py:1883/2969`).
2. **Graph-safe allocator**: a private memory pool + "no allocation during capture" (libtorch `graph_pool_handle` /
   caching-allocator capture mode). The native runtime must replicate this or capture corrupts/fails.
3. **Output lifetime**: clone/detach static outputs before the next replay (`cudagraph_encoder.py:15-17`).
4. **Fail-closed to eager** on shape mismatch / uncaptured B (`:578-599`, `server.py:2080-2084`).

## TO-BE-MEASURED (GPU; BLOCKED until a 5090/L4/L40S run)

- Option B's lost-overlap (replay-mutex contention) vs the per-process tail win.
