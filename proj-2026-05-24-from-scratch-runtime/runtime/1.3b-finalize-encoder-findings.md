# 1.3b — finalize encoder substrate: dynamic-T export is INFEASIBLE → padded fixed-T buckets (production-proven)

## Finding
Attempted a single dynamic-T `torch.export` of the finalize encoder step (`keep_all_outputs=True`, drop=2, T dynamic
20–320) so the C++ port would have one AOTI-compilable program for the variable remainder length (`export_finalize_t2a.py`).
**It fails** with `Constraints violated (T)`: the Conformer pre-encode does 8× subsampling (`//2` three times), so the
encoder output frame count is `1 + ((1 + ((1 + T//2)//2))//2)` — floor arithmetic that generates divisibility guards on T.
torch.export's own suggested fix is `T = 8*_T - 3` (a single residue class mod 8). Real finalize T values do NOT share one
class (idx2 T=45=8·6−3 ✓, but T=44/58 ✗; first-chunk T=261), so **one graph cannot cover arbitrary T** — the graph
structure genuinely depends on T mod 8.

## Decision: padded fixed-T buckets (mirror production)
This is exactly why production finalize used **fixed-T buckets with padding** (the finalize CUDA-graph: T_min=42, T_max=60;
memory `finalize-graph-WIN` / `deployment-target-sagemaker`: "padded finalize SHIPPED byte-exact"). The C++ finalize encoder
substrate should do the same:
- Export/AOTI-compile the finalize encoder at a FIXED chunk T per bucket (fixed-T export is byte-exact — same as the steady
  `export_t2a.py` at T=42/25), with the **valid length passed as a separate input** (so padding is masked, NOT derived from
  the padded chunk shape — the current `export_finalize_t2a.py` derives length internally, which only works for the no-pad case).
- At finalize: pad the remainder `chunk_mel` up to the next bucket T, pass the true `valid_length`, run the bucket graph,
  slice the output to `enc_len`. Common continuation finalize (drop=2) lands in T≈42–64; the rare drop=0 first-chunk
  (T up to ~261) needs a larger bucket or an eager fallback.
- Validate token-benign / WER-neutral exactly like the steady encoder (E.1/E.2): the padded-bucket output, decoded, must
  match the eager finalize tokens.

## Pad-to-single-bucket is ALSO out (not token-safe)
Tested (`test_finalize_padding.py`) padding the finalize chunk to a single fixed bucket T=64 + true length + slice, then
the FULL final-token continuation decode (fork decoder state) vs the unpadded exact-T encoder, over 40 drop=2 finalize
samples: **39/40 token-exact, 1 flip** (idx=18: 62→61, a deletion). The padding is NOT byte-transparent — the depthwise
conv right-context sees the trailing zeros, so the last valid frames drift (enc_out diff **up to 3.6e-1**, erratic). The
output caches are destroyed (diff 27–83) but those are discarded (speculative fork), so only enc_out matters — and at
~2.5% utterance flip rate it's far worse than the steady encoder's 1/1000. **Conclusion: finalize needs EXACT-T encoder
processing (no padding bleed).**

## Remaining options for an EXACT-T finalize encoder in C++ (architectural fork)
1. **Per-exact-T AOTI buckets** (production-faithful: the finalize T range is naturally ~42–60 = pre(9)+remainder+padding;
   plus the rare drop=0 first-chunk outlier). Each fixed-T AOTI is byte-exact (export_t2a proved fixed-T byte-exact). BUT
   each package embeds ~2.5 GB of weights → ~19 packages is impractical UNLESS AOTI **shared weights / constants-on-disk**
   works (it segfaulted on a fresh load in the steady E.3 attempt — would need fixing).
2. **Eager finalize encoder** (finalize is off the hot path, once/turn, so it need not be AOTI-compiled). BUT "eager in C++"
   has no easy callable — cache_aware_stream_step isn't TorchScript-scriptable, and we have no eager conformer in C++. Would
   need either a scripted/exported eager path or a hybrid (native steady + Python/eager finalize), which dents "pure native".
3. **Hybrid**: native steady hot path + a thin eager/Python finalize service (pragmatic; finalize is rare).

## Status / scope
- This is a SEPARATE sub-component from the C++ finalize ORCHESTRATION (state machine + fork isolation + decode-continuation),
  which is validated against the fixture's precomputed eager `enc_out` and does NOT need this substrate (1.3b Phase A).
- `export_finalize_t2a.py` is retained as the record of the dynamic-export dead-end; the padded-bucket exporter is the
  follow-up (1.3b Phase B encoder). Not novel risk — production already runs this; it's implementation + the masking-padding
  byte-exactness check.
