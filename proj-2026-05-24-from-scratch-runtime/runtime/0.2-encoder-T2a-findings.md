# 0.2 / T2a — encoder export byte-exactness (RESOLVED: byte-exact within a fixed geometry)

## Result
**The steady-encoder TorchScript trace is BYTE-EXACT vs eager NeMo** (`validate_encoder_v2.py`): all 5 outputs
(enc_out, enc_len, cache_ch, cache_t, cache_ch_len) `max_abs_diff = 0.000e+00`, with a properly-matched STEADY geometry
(first chunk populates the cache → steady chunk with the populated cache + `drop_extra=2`). T0 eager run-to-run
determinism also holds.

## The subtlety (and the corrected round-3 concern)
The encoder's **output frame-count is DATA-DEPENDENT on cache geometry** (empty/first-chunk vs populated/steady) AND on
`att_context_size`. A first (wrong) comparison showed eager `[1,1024,14]` vs `.ts [1,1024,2]` — but that was a **setup
mismatch** (the `.ts` had been traced with `att_context=[70,1]` + an empty initial cache; the comparison used a
different att-context + an unrealistic empty-cache-with-`drop=2` combo), **NOT trace infidelity.** With matching
geometry the trace is byte-exact.

**So:** the round-3 caveat ("cache-dependent control flow doesn't trace faithfully across geometries") is half-right —
a single trace is **geometry-specific** (frozen att-context + cache-shape + drop-mode), but **WITHIN a geometry the
TorchScript trace reproduces eager byte-for-byte.** Unlike a free-form claim that the encoder can't be exported, the
real requirement is simply: **export the right per-geometry buckets** (first-chunk `drop=0`, steady `drop=2`, finalize
`keep_all_outputs=True` at T_max) with realistic cache inputs — exactly the per-T/per-geometry bucket model the existing
`cudagraph_encoder.py` already uses.

## Implication for the runtime (Phase 1.2b)
- The C++ steady path can load a per-geometry steady-encoder `.ts` and get byte-exact encoder output + cache updates.
- Must export/capture **per geometry**: first-chunk (drop=0), steady (drop=2), finalize (keep_all_outputs, T_max bucket).
  This mirrors the Python server's existing bucketing — not new risk, just per-geometry discipline.
- The att-context must be pinned to the deployed `[70,1]` at export (as `export_encoder.py` does).
- **Encoder byte-exactness (B1a / T2a) is CONFIRMED achievable** — combined with the byte-exact decode (1.2a) and the
  native preprocessor (0.8, next), the full steady path can be byte-exact.

## Note
`encoder_steady_b1.ts` in `microbench/artifacts/` was traced with the EMPTY initial cache (a first-chunk-shaped
geometry, att_context [70,1]) — fine for the 0.1b throughput bench (cost-representative), but the runtime needs the
correctly-bucketed steady/finalize exports per above.
