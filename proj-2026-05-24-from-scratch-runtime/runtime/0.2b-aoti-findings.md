## Toolchain finding — AOTI `.so` exec-stack vs hardened hosts (fixed in the BUILD)
The container-built `wrapper.so` defaulted to `PT_GNU_STACK = RWX` (PF_X set): some linked input (an embedded
constants/cubin blob) lacks a `.note.GNU-stack`, so `ld` marks the output stack executable. (Root cause asserted from the
build behaviour, NOT yet proven with `readelf -S` on each input object — the earlier `codecache.py:184` citation does
not hold in the installed torch and is withdrawn; a reproducible root-cause appendix is TODO.) The **host kernel (6.14,
hardened) refuses to `dlopen`** it: *"cannot enable executable stack as shared object requires: Invalid argument."*
(Loads fine in-container, where the production runtime runs — so this is build hygiene + dev-host validation, not a
runtime blocker.)

**Build fix (not a binary patch):** `aot_compile.py:_force_noexecstack_on_link()` monkeypatches
`torch._inductor.cpp_builder.CppBuilder.get_command_line` to append `-Wl,-z,noexecstack` **only to the shared-lib link**
(gated on `-shared` so Inductor's compile/ISA probes are untouched). `-z noexecstack` forces the output `PT_GNU_STACK`
non-exec regardless of the blob's missing note. Result: `p_flags=6` (RW, non-exec) → host `dlopen` succeeds, and the
byte-exact comparison **reproduced identically** (enc_out 1.99e-5, cache_t 1.66e-2) so the finding is packaging-invariant.
- *Rejected alternatives:* (a) post-hoc clearing PF_X in the `.pt2` = a binary hack (user: "fix the build, not the
  binary"). (b) `package_constants_in_so=False` (constants on disk) shrank the `.so` to 1.2 MB but did **NOT** clear the
  exec flag (the cubins, not the constants, carry it) **and segfaulted on a fresh `aoti_load_package`** → reverted.
- *Inductor has no public ldflags knob* (`aot_inductor.*` config has no link-flag field; `cpp` config namespace is
  opaque) — the `get_command_line` shim is the least-invasive build-time hook. **Deploy implication:** AOTI artifacts
  for hardened hosts must be built with this flag.
