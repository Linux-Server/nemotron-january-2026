# Action D ‚Äî AOTI-compile the T2a encoder ‚Üí C++ runtime: FINDINGS

**Goal (worksofar-FOLDED action D):** AOTInductor-compile the T2a steady-encoder `torch.export` program to a `.so`, load it
from a native (C++) runtime, and byte-exact-compare ‚Äî to close the "T2a (byte-exact encoder) in a C++ RUNTIME is
UNPROVEN" gap. Container `nemotron-aoti:cu128` (torch 2.8.0+cu128, nvcc, sm_120) is the glibc-compatible build env.

## Result ‚Äî the byte-exact bar (T2a) FAILS; the ship bar (T1) PASSES

Compiled `artifacts/enc_steady_t2a.pt2` (the byte-exact `torch.export` program) via `torch._inductor.aoti_compile_and_package`
‚Üí `enc_steady_aoti.pt2` (`aot_compile.py`, in-container). Ran on the real steady-chunk fixture (`t2a_io.pt`), compared
all 5 outputs vs the eager reference:

| output | byte-equal | max_abs_diff |
|---|---|---|
| enc_out | **No** | 1.99e-05 |
| enc_len | yes | 0 |
| cache_ch (last-channel cache) | **No** | 7.19e-05 |
| **cache_t (last-time cache, RECURRENT)** | **No** | **1.66e-02** |
| cache_ch_len | yes | 0 |

**AOTI is NOT byte-exact.** AOTInductor lowers the graph through Inductor/Triton codegen, which **fuses and reorders fp
ops** (different reduction order, FMA/TF32-eligible contractions) vs eager ‚Äî so it diverges. Worst on the **recurrent
`cache_t`** (1.66e-2), which threads forward and accumulates across chunks. The `torch.export` ExportedProgram is
byte-exact **only when run eagerly** (`export_t2a.py`: 0.000e+00); **compiling it breaks byte-equality.**

**This settles T2a for any compiled path, in any language.** The C++ `AOTIModelPackageLoader` would `dlopen` the SAME
`.so` ‚Üí bit-identical numerics to the Python `aoti_load_package`. So **T2a (byte-exact encoder) is unreachable via
AOTI‚ÜíC++** ‚Äî building the C++ loader cannot change this. T2a stays an eager-only / aspirational property.

### But T1 (token-exact, the actual SHIP gate) holds
`aoti_t1_check.py`: streaming loop with the AOTI runner on 19/20 steady chunks (first chunk eager; export is specialized
to the 9+16=25-frame steady geometry), decoded with the verified `ref_greedy_range`, vs all-eager:

```
[all-eager]   0 via steady_fn + 20 eager -> 11 tok  'How much juice is in one lime'
[aoti-steady] 19 via steady_fn +  1 eager -> 11 tok  'How much juice is in one lime'
=== AOTI steady-encoder T1 (token-exact vs all-eager): True ===
```

The 1.66e-2 `cache_t` drift is **token-benign on this clip** ‚Üí AOTI is a **viable encoder vehicle at the T1 bar**. (Per
the de-banking discipline: ONE clip ‚âÝ parity. The real T1 gate is full-1000 WER-within-CI ‚Äî action E/F ‚Äî not a per-clip
token match. This probe only shows AOTI is not *obviously* token-divergent.)

## Toolchain finding ‚Äî AOTI `.so` exec-stack vs hardened hosts (fixed in the BUILD)
The container-built `wrapper.so` defaulted to `PT_GNU_STACK = RWX` (PF_X set): the embedded constants/cubin blob has no
`.note.GNU-stack`, so `ld` marks the output stack executable. The **host kernel (6.14, hardened) then refuses to
`dlopen`** it: *"cannot enable executable stack as shared object requires: Invalid argument."* (Loads fine in-container,
which is where the production runtime runs ‚Äî so this is build hygiene + dev-host validation, not a runtime blocker.)

**Build fix (not a binary patch):** `aot_compile.py:_force_noexecstack_on_link()` monkeypatches
`torch._inductor.cpp_builder.CppBuilder.get_command_line` to append `-Wl,-z,noexecstack` **only to the shared-lib link**
(gated on `-shared` so Inductor's compile/ISA probes are untouched). `-z noexecstack` forces the output `PT_GNU_STACK`
non-exec regardless of the blob's missing note. Result: `p_flags=6` (RW, non-exec) ‚Üí host `dlopen` succeeds, and the
byte-exact comparison **reproduced identically** (enc_out 1.99e-5, cache_t 1.66e-2) so the finding is packaging-invariant.
- *Rejected alternatives:* (a) post-hoc clearing PF_X in the `.pt2` = a binary hack (user: "fix the build, not the
  binary"). (b) `package_constants_in_so=False` (constants on disk) shrank the `.so` to 1.2 MB but did **NOT** clear the
  exec flag (the cubins, not the constants, carry it) **and segfaulted on a fresh `aoti_load_package`** ‚Üí reverted.
- *Inductor has no public ldflags knob* (`aot_inductor.*` config has no link-flag field; `cpp` config namespace is
  opaque) ‚Äî the `get_command_line` shim is the least-invasive build-time hook. **Deploy implication:** AOTI artifacts
  for hardened hosts must be built with this flag.

## Decision impact
- **Encoder correctness bar for the native runtime = T1, not T2a.** Any compiled encoder (AOTI today; TensorRT/hand
  kernels later) will be non-bit-exact; the ship gate is WER-within-CI + sequencing equivalence (already the pinned T1
  bar ‚Äî `0.4-decision-FINAL`). T2a was "aspirational"; this confirms it stays Python-eager-only.
- **AOTI is a usable encoder backend** for the C++ runtime (token-benign here) ‚Äî but it is NOT a free byte-exact win;
  it must be validated at the T1/WER bar on the corpus, and the `cache_t` accumulation must be watched over long streams.
- **C++ `AOTIModelPackageLoader` wiring is now plumbing, not a numeric gate** (numerics proven identical via the shared
  `.so`). Folds into Phase-1 1.3/1.4 (the session/finalize C++), built in-container with `noexecstack`.
