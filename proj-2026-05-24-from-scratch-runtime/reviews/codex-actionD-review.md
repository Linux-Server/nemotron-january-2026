# Adversarial review: action D AOTI findings

1. [BLOCKER] Action D is not closed; the native C++ loader was skipped.
   The accepted action was explicit: "AOTI-compile the T2a encoder in the container -> C++ byte-exact vs eager" (`reviews/worksofar-FOLDED.md:65`) and the phase table says "load in C++ + byte-exact compare" with `runtime/cpp/` as the file target (`runtime/PHASE1-STEPS.md:25`). The new writeup restates that goal (`runtime/0.2b-aoti-findings.md:3-5`) and then replaces it with an argument that C++ would `dlopen` the same `.so` (`runtime/0.2b-aoti-findings.md:26-28`, `runtime/0.2b-aoti-findings.md:68-69`). That is a goalpost move, not closure. Recommended change: keep action D open until a minimal C++ harness using `torch::inductor::AOTIModelPackageLoader` loads `enc_steady_aoti.pt2`, feeds `t2a_io.pt`, and compares all outputs against the eager fixture byte-for-byte or records the exact non-byte-exact failure.

2. [BLOCKER] The result fails the stated T2a/B1a encoder gate and is being reframed without an explicit decision.
   The decision memo's T1 ship gate is full-1000 WER plus exact event/final-delta/generation behavior, while T2a is still the encoder byte-exact fixture gate (`reviews/0.4-decision-FINAL.md:23-26`). The phase plan still lists the AOTI wire gate as "C++ byte-exact vs eager" (`runtime/PHASE1-STEPS.md:25`). The writeup reports `cache_t` max drift `1.66e-02` (`runtime/0.2b-aoti-findings.md:13-24`) and then says the native encoder correctness bar is now T1 (`runtime/0.2b-aoti-findings.md:62-65`). Recommended change: state plainly that action D failed the T2a byte-exact objective. If the project wants to continue with AOTI as a T1-only encoder, require an explicit risk sign-off and move the corpus/event-stream T1 gate ahead of further runtime banking.

3. [BLOCKER] "T1 passes" is over-banked from one short utterance with 19 compiled steady chunks.
   The only T1 probe uses cached dataset item `ds[1]` (`runtime/aoti_t1_check.py:66-68`), one streaming loop, and compares a single token list (`runtime/aoti_t1_check.py:84-89`). The writeup admits one clip is not parity (`runtime/0.2b-aoti-findings.md:40-42`) but still calls AOTI a "viable encoder vehicle at the T1 bar" (`runtime/0.2b-aoti-findings.md:40`, `runtime/0.2b-aoti-findings.md:66-67`). The hidden failure mode is exactly the dangerous one: recurrent `cache_t` drift accumulates silently until a later near-tie flips, deletes, or inserts a token; the runtime will not notice because the compiled path no longer has a byte-exact invariant. Recommended change: downgrade the claim to "one smoke clip is token-exact." Add long-stream and full-1000 runs, per-chunk drift curves, argmax-margin reports, and explicit late-stream deletion/flip checks before calling it T1-viable.

4. [MAJOR] The numeric-drift measurement is one steady-step fixture, not a streaming drift characterization.
   `aot_compile.py` loads `t2a_io.pt`, runs the package once, and compares that single input/output tuple (`runtime/aot_compile.py:39-54`). The fixture itself is saved from chunk 1 only (`runtime/export_t2a.py:76-81`), even though the eager export validation loops over the one clip's 20 chunks (`runtime/export_t2a.py:54-73`). That means `1.66e-02` is not a proven worst case across the trajectory, across cache histories, or across long streams. Recommended change: add an AOTI streaming byte-diff harness that runs every steady chunk against the eager step while feeding each path's own recurrent caches, and report max/mean drift per chunk plus token margins.

5. [MAJOR] "T2a is unreachable via AOTI in any language" is not supported by the experiment.
   The writeup generalizes from one default AOTI compile to "any compiled path, in any language" (`runtime/0.2b-aoti-findings.md:26-28`) and "any compiled encoder" (`runtime/0.2b-aoti-findings.md:63-65`). Current `aot_compile.py` calls `torch._inductor.aoti_compile_and_package(ep, package_path=pkg_path)` with no accuracy-oriented `inductor_configs` or precision/determinism flags (`runtime/aot_compile.py:33-36`), and the searched files show no attempted TF32, deterministic, fusion, fallback, or autotune matrix. Recommended change: replace the universal claim with "default AOTI failed byte-exact." Try and log a small compile matrix: TF32 disabled, deterministic settings, fusion/autotune reductions where available, and any AOTI fallback/strict modes. If those do not recover byte equality, document the failed knobs and exact generated-kernel evidence.

6. [MAJOR] "Same `.so`" ignores real Python-vs-C++ runtime variables.
   Python's `AOTICompiledModel.__call__` flattens tensors and calls `loader.boxed_run(flat_inputs)` with no stream handle (`/usr/local/lib/python3.12/dist-packages/torch/export/pt2_archive/_package.py:429-436`). The C++ package loader exposes `run` and `boxed_run` with an optional `stream_handle` (`/usr/local/lib/python3.12/dist-packages/torch/include/torch/csrc/inductor/aoti_package/model_package_loader.h:22-29`), explicit constant loading/update APIs (`/usr/local/lib/python3.12/dist-packages/torch/include/torch/csrc/inductor/aoti_package/model_package_loader.h:32-43`), and CUDA runner APIs with device strings and explicit CUDA streams (`/usr/local/lib/python3.12/dist-packages/torch/include/torch/csrc/inductor/aoti_runner/model_container_runner_cuda.h:15-31`). Those are exactly the seams where allocator, stream ordering, device index, input contiguity, output ownership, and user-managed constants can differ. Recommended change: the C++ harness must test default stream and explicit stream, device index, dtype/device/stride/contiguity, output aliasing/lifetime, and any constants-on-disk path before calling C++ "plumbing."

7. [MAJOR] The build/noexec story is internally inconsistent and the artifact section is stale.
   The writeup says the build fix is `_force_noexecstack_on_link()` (`runtime/0.2b-aoti-findings.md:50-54`), while the artifact section still says `enc_steady_aoti.pt2` is the RWX-stack build and `enc_steady_aoti_noexec.pt2` is the host-loadable patched artifact (`runtime/0.2b-aoti-findings.md:71-73`). At review time only `enc_steady_aoti.pt2` exists in `runtime/artifacts/`, and it host-loads successfully; there is no `enc_steady_aoti_noexec.pt2`. Recommended change: update the artifact section to match the actual artifact, include `zipinfo`/`readelf -l`/host `aoti_load_package` output, and remove stale references to a post-patched noexec artifact unless it is actually kept.

8. [MAJOR] The constants-on-disk history is not clean enough to support packaging-invariant conclusions.
   The current script embeds constants via the default compile path (`runtime/aot_compile.py:33-36`) and the writeup says `package_constants_in_so=False` was rejected because it did not clear execstack and segfaulted (`runtime/0.2b-aoti-findings.md:55-57`). That is a materially different story from "constants-on-disk fixed noexecstack." The installed Torch config keys do exist (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/torch/_inductor/config.py:1382-1386`), but the current script does not use them. Recommended change: record two separate build rows: embedded-constants + `-z noexecstack`, and constants-on-disk. For each, log load success/failure, byte-diff results, package contents, and whether the `.so` has executable stack.

9. [MAJOR] The noexec root-cause citation is not verified by the evidence presented.
   The current comments claim a constants/cubin blob without `.note.GNU-stack` causes RWX stack (`runtime/aot_compile.py:11-17`), while the writeup says the cubins, not constants, carry it (`runtime/0.2b-aoti-findings.md:55-57`). Against installed Torch 2.8.0, the earlier `codecache.py:184` citation is not a noexecstack flag; it just returns `"cubin"` for CUDA (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/torch/_inductor/codecache.py:182-184`). Recommended change: add a reproducible root-cause appendix with the exact linker command, `readelf -S` on every linked object/blob, `readelf -l` on the output `.so`, and the before/after `PT_GNU_STACK` flags. Do not cite `codecache.py:184` for noexecstack.

10. [MAJOR] The monkeypatch is private, global, and unguarded.
    `_force_noexecstack_on_link()` monkeypatches `torch._inductor.cpp_builder.CppBuilder.get_command_line` and relies on private `_do_link`, string `-shared`, and command concatenation (`runtime/aot_compile.py:18-25`). That may be acceptable as a local workaround, but it is not production build infrastructure until it fails closed. Recommended change: assert that the generated shared-library link command contains `-Wl,-z,noexecstack`, fail the build if it does not, print the command into the artifact log, and isolate this in a version-pinned build wrapper instead of leaving an implicit process-global patch.

11. [MAJOR] The steady-only AOTI probe does not cover required runtime geometries.
    `aoti_t1_check.py` explicitly keeps the first chunk and non-25-frame tail eager because the export is specialized to the 9+16=25 steady geometry (`runtime/aoti_t1_check.py:4-6`, `runtime/aoti_t1_check.py:38-43`). The phase plan specifically says to "Export first-chunk geometry too" for the AOTI wire step (`runtime/PHASE1-STEPS.md:25`), and finalize is still a separate Phase 1 gate (`runtime/PHASE1-STEPS.md:26`). Recommended change: do not fold this into session/finalize work until first, steady, and finalize/`keep_all_outputs` packages have native-load tests and a clear fallback policy for tails.

    The authoritative work-so-far action list still requires incremental-STFT parity, a corpus fixture matrix, and a full-1000 T1 shadow run (`reviews/worksofar-FOLDED.md:66-67`). The decision memo defines T1 as full-1000 WER within CI plus exact per-session event/final-delta/generation equivalence (`reviews/0.4-decision-FINAL.md:23-26`). Recommended change: mark AOTI as "candidate backend, blocked on E/F" and prohibit downstream "usable encoder backend" language until the full T1 corpus and event-stream gate actually passes.

13. [MAJOR] The one utterance is not even the full fixture transcript boundary.
    The T1 probe reports 11 tokens and text without the question mark (`runtime/0.2b-aoti-findings.md:34-37`), while the fixture manifest for the same sentence has 12 tokens and the reference includes `?` (`runtime/fixtures/manifest.json:74-98`). 14. [MINOR] The comparison harness omits metadata that matters for C++ integration.
    `aot_compile.py` checks shape equality, `torch.equal`, and max absolute diff (`runtime/aot_compile.py:48-54`), but it does not log dtype, device, stride, contiguity, storage aliasing, call spec, model metadata, package hash, or generated wrapper name. Recommended change: extend the fixture comparison to record dtype/device/stride/contiguity for inputs and outputs, output aliasing/lifetime, package SHA256, wrapper `.so` path, Torch/CUDA versions, and call spec.
