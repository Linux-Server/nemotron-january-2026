# Codex Step 5 Hardening Review

1. [BLOCKER] `vad_start_cancel=PASS` is not a real vad-start cancellation test.

   Evidence: the reference behavior being claimed is not just `PENDING_FINALIZE -> STREAMING`. While pending, new audio is held in `continuous_post_stop_audio` (`runtime/finalize_ref.py:398`-`runtime/finalize_ref.py:402`), `vad_start()` cancels debounce and flushes that held audio (`runtime/finalize_ref.py:425`-`runtime/finalize_ref.py:430`), and the canary asserts parent unchanged while pending, held audio flushed, equality with a no-stop session, next steady, and final tokens (`runtime/finalize_ref.py:1223`-`runtime/finalize_ref.py:1267`). The C++ synthetic coverage path does none of that: it snapshots a fresh empty session, sets `mode = PENDING_FINALIZE`, immediately sets `mode = STREAMING`, and checks counters stayed equal (`runtime/cpp/session_main.cpp:2977`-`runtime/cpp/session_main.cpp:2988`). There is no held post-stop audio buffer in `SessionState`, and the C++ audio appender rejects non-STREAMING input (`runtime/cpp/session_main.cpp:2183`), so this path cannot exercise the production cancel/flush behavior. The manifest then reports that synthetic flip as lifecycle coverage (`runtime/cpp/session_main.cpp:3022`-`runtime/cpp/session_main.cpp:3029`; reported PASS in `runtime/artifacts/logs/session_multiturn_audio_step5.log:94`).

   Fix: implement a real C++ vad-start-cancel scenario: append pre audio, enter `PENDING_FINALIZE`, append post-stop audio into a held buffer without changing ASR state, cancel to `STREAMING`, flush/drain, assert equality with a no-stop session, append continuation, then finalize and compare tokens/events. Until then, remove `vad_start_cancel_synthetic=PASS` from the coverage manifest.

2. [MAJOR] The preproc manifest is fail-closed for a mismatched hash, but export reuse is still self-referential and incomplete.

   Evidence: the C++ runtime does refuse a simple corrupt-hash case: it requires `preproc.ts.manifest.json` (`runtime/cpp/session_main.cpp:1795`-`runtime/cpp/session_main.cpp:1798`) and throws when `preproc_ts_sha256` differs from `sha256_file(preproc.ts)` (`runtime/cpp/session_main.cpp:1831`-`runtime/cpp/session_main.cpp:1835`). That is good, but the exporter computes the expected hash from the already-existing `preproc.ts` during reuse (`runtime/export_session_bundle.py:476`-`runtime/export_session_bundle.py:480`, `runtime/export_session_bundle.py:366`-`runtime/export_session_bundle.py:382`). A stale `preproc.ts` with a matching stale manifest is therefore accepted if the shallow contract fields match and the one-sample self-check passes. The shipped manifest also has `"preprocessor_config": {}` (`runtime/artifacts/preproc.ts.manifest.json:33`), so it is not actually recording the STFT/window/feature/normalization config requested by the Step 3 fold. C++ ignores `preprocessor_config` entirely and only checks schema/model/trace_k/dither/geometry/hash (`runtime/cpp/session_main.cpp:1802`-`runtime/cpp/session_main.cpp:1835`).

   Fix: either regenerate `preproc.ts` unconditionally for audio exports, or make reuse depend on a non-self-referential contract derived from the loaded model, including the real NeMo preprocessor config (`_cfg` or equivalent), versions, geometry, dither, and trace shape. Add a negative test that corrupts `preproc_ts_sha256` and asserts the C++ binary exits before loading/running audio, analogous to the bucket hash checks (`runtime/cpp/session_main.cpp:1427`-`runtime/cpp/session_main.cpp:1430`).

3. [MAJOR] The determinism gate is same-process with shared loaders; it is not an independent run-to-run proof.

   Evidence: the single-stream rerun resets `SessionState`, but reuses the same already-loaded `enc_first`, `enc_steady`, `finalize_loaders`, `joint`, and `predict` objects for the second pass (`runtime/cpp/session_main.cpp:3904`-`runtime/cpp/session_main.cpp:3919`). The multi-turn rerun does the same (`runtime/cpp/session_main.cpp:3639`-`runtime/cpp/session_main.cpp:3655`). The preproc determinism check runs the same `preproc` module twice in the same process after warmup (`runtime/cpp/session_main.cpp:1985`-`runtime/cpp/session_main.cpp:2003`). This catches accidental mutable `SessionState` leakage and same-process nondeterminism, but it does not catch fresh-process loader initialization differences, cross-process cuFFT plan variation, or stale global loader state. The reported PASS lines are therefore narrower than "run-to-run determinism" (`runtime/artifacts/logs/session_audio_n200_step5.log:265`, `runtime/artifacts/logs/session_multiturn_audio_step5.log:89`).

   Fix: add a process-level determinism gate: run the C++ binary twice from fresh processes, reload all artifacts each time, and compare emitted token/event fingerprints plus the preproc fixed-block output hash. At minimum, make the in-process rerun instantiate fresh AOTI/TorchScript loaders for the second pass and label the current check `same_process_full_session_twice`.

4. [MAJOR] `alias_after_clone=0` proves the clone barrier, not absence of AOTI loader output reuse.

   Evidence: recurrent caches are assigned as clones (`runtime/cpp/session_main.cpp:1662`-`runtime/cpp/session_main.cpp:1664`), and the alias assertion checks the cloned state tensor against the immediate output tensor (`runtime/cpp/session_main.cpp:1665`-`runtime/cpp/session_main.cpp:1673`). That will almost always pass if `clone()` works; it does not prove `out[2..4]` themselves are not loader-reused storage, nor that raw AOTI outputs do not alias inputs or previous raw outputs. The long-stream check is meaningful for the state that will be fed back: it holds cloned cache tensors and verifies they do not mutate after later AOTI calls (`runtime/cpp/session_main.cpp:2833`-`runtime/cpp/session_main.cpp:2863`). But with clone-on-assign enabled, it mostly validates the barrier, not the underlying loader ownership contract. The summary name overclaims this as "AOTI CACHE OWNERSHIP" (`runtime/cpp/session_main.cpp:2735`-`runtime/cpp/session_main.cpp:2742`; reported `alias_after_clone=0` in `runtime/artifacts/logs/session_n1000_step5.log:1066`).

   Fix: either rename the gate to `cache_state_clone_barrier` and document that raw loader output reuse is intentionally tolerated, or add pre-clone checks that compare raw `out[2..4]` against inputs and previous raw outputs across consecutive `run()` calls.

5. [MAJOR] The audio oracle shift is sound for "shipped preproc.ts runtime", but it weakens the eager-model gate.

   Evidence: in audio export, `RecordingContinuousFinalizeRef` uses `preproc.ts` for the recorded mels/events (`runtime/export_session_bundle.py:1029`-`runtime/export_session_bundle.py:1036`), and `main()` explicitly prints that the audio bundle oracle is `preproc.ts` mels/events (`runtime/export_session_bundle.py:1802`-`runtime/export_session_bundle.py:1804`). C++ then compares its own `preproc.ts` output against those bundle mels (`runtime/cpp/session_main.cpp:2115`-`runtime/cpp/session_main.cpp:2119`, `runtime/cpp/session_main.cpp:2317`-`runtime/cpp/session_main.cpp:2326`). The eager path is only a separate CI envelope (`runtime/export_session_bundle.py:681`-`runtime/export_session_bundle.py:804`), and the actual Step 5 artifacts show tolerance-based, non-byte-equal bundle drift (`runtime/artifacts/session_audio_bundle.ts.audio_ci.json:21`-`runtime/artifacts/session_audio_bundle.ts.audio_ci.json:35`). So the token/event exact gate is now C++ vs shipped TS preproc, not C++ vs eager model.

   Fix: keep this oracle if the claim is "C++ runtime matches the shipped preproc.ts artifact," but add a separate eager semantic gate: eager-preproc tokens/events for the same audio rows, or a documented TS-vs-eager token/event/WER non-regression check. Do not summarize the audio result as eager-equivalent merely because the one-sample export self-check is byte-equal (`runtime/export_session_bundle.py:432`-`runtime/export_session_bundle.py:463`).

6. [MINOR] The first-chunk path is honestly labeled, but the margin number is only an observational corpus margin.

   Evidence: the runtime still loads and executes `enc_first.ts` (`runtime/cpp/session_main.cpp:3305`-`runtime/cpp/session_main.cpp:3310`, `runtime/cpp/session_main.cpp:1679`-`runtime/cpp/session_main.cpp:1690`), and the summary explicitly says choice (b) retained TorchScript and dropped the unqualified pure-native claim (`runtime/cpp/session_main.cpp:2705`-`runtime/cpp/session_main.cpp:2728`; `runtime/artifacts/logs/session_n1000_step5.log:1065`). The drift measurement is real for `enc_first.ts` enc_out vs eager enc_out (`runtime/cpp/session_main.cpp:1693`-`runtime/cpp/session_main.cpp:1715`). The margin is also real for executed greedy decisions: it is `top1 - top2` from the joint logits (`runtime/cpp/session_main.cpp:1612`-`runtime/cpp/session_main.cpp:1620`) and is recorded only on the expected-first path (`runtime/cpp/session_main.cpp:2036`-`runtime/cpp/session_main.cpp:2058`). But `margin_checks=2000` for 1000 rows means this is basically the first two emitted encoder frames per utterance, not a global proof that `~6e-5` encoder drift can never flip a later or alternate decode path.

   Fix: keep the current wording narrow: "TorchScript first chunk retained; first-chunk drift observed; tested first-chunk greedy decisions have min margin 17." A pure-native runtime still requires an AOTI first-chunk path.

7. [MINOR] `FORK_ASSERT` now covers mode and audio counters, but the label still conflates fork isolation with post-finalize lifecycle.

   Evidence: parent snapshots now include mode, raw audio, pending audio, total audio samples, and synthetic prefix samples (`runtime/cpp/session_main.cpp:968`-`runtime/cpp/session_main.cpp:984`), and `fork_assert_parent_unchanged()` checks those fields (`runtime/cpp/session_main.cpp:987`-`runtime/cpp/session_main.cpp:1023`). The long-lived multi-turn path resets once per stream (`runtime/cpp/session_main.cpp:3413`-`runtime/cpp/session_main.cpp:3415`), runs both turns in the same session (`runtime/cpp/session_main.cpp:3420`-`runtime/cpp/session_main.cpp:3514`), and checks retained audio counters/ring after each speculative finalize (`runtime/cpp/session_main.cpp:2523`-`runtime/cpp/session_main.cpp:2544`), then true-boundary cold reset (`runtime/cpp/session_main.cpp:3581`-`runtime/cpp/session_main.cpp:3608`). The per-row reset still exists for the single-stream corpus (`runtime/cpp/session_main.cpp:3740`-`runtime/cpp/session_main.cpp:3742`), but the multi-turn gate is the correct long-lived lifecycle test. The caveat is naming: inside `run_finalize()`, the parent and snapshot are both moved to `FINALIZED` before cloning (`runtime/cpp/session_main.cpp:2353`-`runtime/cpp/session_main.cpp:2356`), so `FORK_ASSERT` itself proves parent immutability during fork flush; retained/cold-reset checks prove the post-finish lifecycle.

   Fix: split the printed counters into `fork_parent_unchanged`, `speculative_retained_state`, and `true_boundary_cold_reset`, or keep `FORK_ASSERT` but document that lifecycle correctness is proven by the retained/cold-reset checks, not that one flag alone.

## Verdict

Step 5 hardening is **NOT MET**. The first-chunk disclosure is honest, long-lived multi-turn retained-state assertions are real, and the C++ preproc hash check is fail-closed for simple mismatches. But the adversarial coverage manifest has a false PASS for `vad_start_cancel`, the determinism proof is only same-process, provenance reuse can still bless a stale self-consistent `preproc.ts`, and several summaries overclaim what the clone and audio-oracle checks prove.
