# Codex Adversarial Work-So-Far Review

Scope reviewed: `PLAN.md`, `reviews/decision.md`, `reviews/0.4-decision-FINAL.md`, `reviews/cppstream-FOLDED.md`, the requested `runtime/` docs/code, and the 0.1b microbench artifacts.

## Findings

1. [BLOCKER] The L40S density gate overclaims production relevance because the "decode" model is a pure host sleep that overlaps the encoder graph instead of depending on its output. `RESULTS-L40S.md` explicitly says the L40S gate used mock host sleep `--decode-host-us 10000` calibrated to thread-busy time (`spikes/0.1-overlap-ablation/microbench/RESULTS-L40S.md:3-7`). In code, the lane replays the encoder graph asynchronously, records an event, then the worker sleeps for decode host time, and only then waits for the GPU (`spikes/0.1-overlap-ablation/microbench/microbench.cpp:121-128`, `spikes/0.1-overlap-ablation/microbench/microbench.cpp:180-186`). Real RNNT decode is a sequence of joint/predict GPU calls plus host argmax synchronizations (`runtime/export_decode.py:30-37`, `runtime/cpp/steady_main.cpp:46-55`) and it cannot generally start until the encoder output exists. This benchmark can turn encoder time into "free overlap" that the real dependency chain may not have. Recommendation: rerun the L40S gate with an encoder-event-dependent decode chain using the exported joint/predict modules, or at minimum run `decode_gpu_iters` and host-time sensitivity where decode work is ordered after encoder completion; do not call 16->32-40 production density banked until that passes.

2. [BLOCKER] The final GO decision treats finalize sensitivity as cleared, but the only deploy-target L40S result is steady-only and explicitly says finalize is missing. The final decision table says finalize sensitivity is "YES" based on 5090 lanes=1 ~12 vs lanes=8 >=24 (`reviews/0.4-decision-FINAL.md:29-32`). The L40S gate result says "NO finalize path yet" and says finalizes would lower the multiplier and still need confirmation (`spikes/0.1-overlap-ablation/microbench/RESULTS-L40S.md:34-39`, `spikes/0.1-overlap-ablation/microbench/RESULTS-L40S.md:45-48`). The 5090 finalize model is only periodic extra graph replays plus a host sleep, not a real `keep_all_outputs` finalize bucket (`spikes/0.1-overlap-ablation/microbench/RESULTS-5090.md:41-57`). Recommendation: downgrade the 0.4 GO to "conditional GO pending L40S finalize sensitivity" or rerun L40S with a faithful finalize bucket before using the >=1.5x gate as cleared.

3. [BLOCKER] The "GPU-bound at ~32-40" conclusion is not robust because it is measured under the same synthetic decode model whose GPU load is zero. L40S lanes=8 and lanes=12 look equivalent at 32/48 streams (`spikes/0.1-overlap-ablation/microbench/RESULTS-L40S.md:17-23`), and the doc turns that into "ceiling is the real encoder compute, not the lanes" (`spikes/0.1-overlap-ablation/microbench/RESULTS-L40S.md:28-29`). But the benchmark default has `decode_gpu_iters = 0` (`spikes/0.1-overlap-ablation/microbench/microbench.cpp:52-54`), while real decode repeatedly launches `joint` and `predict` and syncs token decisions (`runtime/cpp/decode_main.cpp:18-28`). Recommendation: report this as "GPU-bound for encoder replay plus host-sleep decode"; require an L40S sweep with realistic decode GPU work before concluding the native ceiling is encoder-bound.

4. [MAJOR] The microbench does not actually satisfy the original 0.1 spec/topology matrix, so it is not a full conjunct-2 proof. The spec asked for single-process/single-context vs MPS vs multi-process, per-lane stream/event dependency, 5090 and L40S, and finalize (`spikes/0.1-overlap-ablation/0.1b-microbench-spec.md:29-40`). The plan similarly says 0.1 should compare single-context vs MPS and report CUDA-event timelines and queue/lane wait, not only throughput knee (`PLAN.md:470-478`). The implemented result is one-process lane-count A/B with a chunk-latency proxy; `0.1a` remains pending in the progress table (`PLAN.md:653-654`). Recommendation: split the claim: 0.1b is a useful single-process replay signal, not proof that the production topology will beat K=3 Python/MPS under real scheduler/finalize behavior.

5. [MAJOR] The density claim also assumes shared weights, but the L40S bench hit the opposite memory shape. The L40S report says lanes=32 OOM'd because each lane loaded its own 2.5 GB model copy and that a real runtime needs shared read-only weights (`spikes/0.1-overlap-ablation/microbench/RESULTS-L40S.md:40-43`). The state/mutability plan still treats shared weights as a future risk area, not an implemented fact (`PLAN.md:585-586`, `PLAN.md:650-651`). Recommendation: make "32-40 streams in one process" conditional on a successful shared-weight/graph-buffer design; otherwise the measured lane topology is not the production memory topology.

6. [BLOCKER] The C++ streaming "full-chunk-only" fix is not server-faithful until finalize exists; today it defers or drops the exact audio region that decides final semantics. `steady_main.cpp` processes only complete 16-frame chunks and records the remainder as "-> finalize" (`runtime/cpp/steady_main.cpp:88-90`). The non-multiple-of-16 robustness test only verifies "NO CRASH"; it does not compare transcript, final delta, or fork/resume behavior (`runtime/cpp/steady_main.cpp:97-106`). The phase plan confirms C++ finalize and the state machine are still future work (`runtime/PHASE1-STEPS.md:24-25`). Recommendation: do not call the steady loop T1 server-faithful for non-multiple clips; add a finalize implementation and parity tests for `Tm = 1,15,16,17,31,319,320,321` before closing the partial-chunk blocker.

7. [BLOCKER] Streaming preprocessing is not byte-exact to the server path; the validation slices a full-clip mel instead of exercising incremental STFT/ring behavior. `stream_decode.py` computes `mel` once for the whole audio (`runtime/stream_decode.py:37-39`) and then slices `new_mel = mel[:, :, pos:pos+shift]` (`runtime/stream_decode.py:52-58`). `export_stream_encoder.py` does the same and even asserts the selected clip is a multiple of `shift` (`runtime/export_stream_encoder.py:34-35`). The prior folded review already called this out as offline full-clip mel, not server incremental STFT plus `raw_audio_ring` (`reviews/cppstream-FOLDED.md:28-30`). `validate_preproc.py` only tests fixed-size random chunks and a TorchScript export, not across-chunk STFT continuity (`runtime/validate_preproc.py:22-31`, `runtime/validate_preproc.py:33-47`). Recommendation: implement and validate incremental preprocessing against the Python server before using any "audio->streaming pipeline byte-exact" language.

8. [MAJOR] The C++ decode byte-exact claim is based on one bundled fixture, not the full fixture set. `export_decode.py` validates all fixtures in Python (`runtime/export_decode.py:62-73`), but the C++ self-test bundle contains only `decode_real_1.pt` (`runtime/export_decode.py:75-89`). `decode_main.cpp` loads only that bundled `enc`, `enc_len`, and `gold` (`runtime/cpp/decode_main.cpp:39-44`). Recommendation: make the C++ test iterate all six fixtures, plus the missing max-symbol and streaming continuation fixtures, before saying "C++ byte-exact" without a qualifier.

9. [MAJOR] The Python "18/18 streaming partial-hyp continuation" check is self-consistency, not equivalence to NeMo's deployed `partial_hypotheses` path. `ref_decode.py` splits a full encoder output into 2/3/5 ranges and checks carried-state decode equals full decode (`runtime/ref_decode.py:47-58`, `runtime/ref_decode.py:77-88`). That is useful, but it does not call `rnnt_decoder_predictions_tensor(... partial_hypotheses=...)`; only the one-clip `stream_decode.py` path does that (`runtime/stream_decode.py:68-81`). Recommendation: rename the 18/18 result to "reference split self-consistency" and add multi-fixture NeMo partial-hypothesis oracle checks.

10. [MAJOR] The "BYTE/STATE-exact" decode bar is not met by the current checks; the evidence mostly compares token sequences. The decode contract lists `score`, `dec_state`, `predictor_state`, `predictor_output`, `label`, `decoded_length`, `last_frame`, timestamps, etc. as part of the native output surface (`runtime/0.6a-decode-findings.md:11-16`). The reported result says byte-identical `y_sequence` on six fixtures (`runtime/0.6a-decode-findings.md:37-43`), and the reference implementation returns only `hyp` tokens plus raw carried tensors (`runtime/ref_decode.py:20-36`). The doc admits per-token predictor-state snapshots and streaming continuation fixtures were missing at that point (`runtime/0.6a-decode-findings.md:27-28`). Recommendation: require state-object equality checks for `LabelLoopingStateItem` fields and emitted event metadata, not just token lists.

11. [MAJOR] The fixture base is too narrow for the byte-exact conclusions. The committed manifest has three empty/noise fixtures with zero tokens (`runtime/fixtures/manifest.json:2-40`) and three real encoder-output fixtures (`runtime/fixtures/manifest.json:41-170`), while the end-to-end audio and streaming paths repeatedly use dataset item `ds[1]`, "How much juice is in one lime?" (`runtime/export_pipeline.py:25-29`, `runtime/stream_decode.py:29-35`, `runtime/export_t2a.py:23-30`). The phase table still turns this into broad "FULL C++ pipeline" and "Python STREAMING pipeline" passes (`runtime/PHASE1-STEPS.md:18-22`). Recommendation: add a corpus-level fixture matrix: varied lengths, non-multiple chunk endings, silence-leading/trailing, fast speech, punctuation, near-tie logits, max-symbol saturation, and at least a full-1000 T1 shadow run before claiming production byte-exactness.

12. [MAJOR] The all-blank noise cases are weak evidence for decode correctness. They exercise blank termination but not predictor state evolution, repeated non-blanks, max-symbol saturation, or partial-hypothesis mutation; the manifest shows all three noise/quiet fixtures have empty `y_sequence` (`runtime/fixtures/manifest.json:2-40`). The 0.6a doc itself lists `max_symbols=10` saturation as missing (`runtime/0.6a-decode-findings.md:27-28`), but the phase table demotes it to a "remaining minor" (`runtime/PHASE1-STEPS.md:13`). Recommendation: treat max-symbol saturation and state snapshots as required, not minor, because that is where label-looping boundary bugs hide.

13. [BLOCKER] T2a is not achieved for the actual C++ runtime; it is achieved for a Python `ExportedProgram` that the runtime cannot load yet. `T2a-findings.md` says `.pt2` cannot be loaded by `torch::jit::load` and must be AOTInductor-compiled to a `.so` (`runtime/T2a-findings.md:21-24`). It also says local AOTInductor is blocked by the glibc/nvcc issue (`runtime/T2a-findings.md:25-27`). The phase plan has a separate pending `1.2b-wire` step to wire T2a into C++ via AOTInductor and export first-chunk geometry (`runtime/PHASE1-STEPS.md:23`). Recommendation: phrase the status as "Python export byte-exact; C++ runtime integration unproven" until AOTI is built, loaded, and compared in C++.

14. [MAJOR] The T2a validation is only steady geometry on one clip and one cache-length trajectory. `export_t2a.py` exports the steady step using `c1` from the selected clip (`runtime/export_t2a.py:38-47`), runs first chunk eagerly rather than through an exported first-chunk graph (`runtime/export_t2a.py:60-64`), and validates the same full-clip-mel trajectory through 19 steady chunks (`runtime/export_t2a.py:58-76`). The findings doc says first-chunk geometry still needs the same treatment (`runtime/T2a-findings.md:15-20`). Recommendation: add first-chunk, finalize `keep_all_outputs`, non-multiple remainder, long clips beyond cache saturation, reset-to-empty-cache, and B>1/session-interleaving T2a checks before claiming "byte-exact streaming encoder" generally.

15. [MAJOR] The current C++ streaming state is a miniature demo state, not the complete server state whose reset/resume semantics must be proven. The design says a `Session` owns encoder cache, decoder hypothesis state, raw audio ring, mel frame ring, pending audio, finalize state machine, and generation counter (`runtime/1.0-state-ownership-design.md:9-17`, `runtime/1.0-state-ownership-design.md:22-35`). `steady_main.cpp` resets only clc/clt/clcl, g/h/c, ring, emitted, and `hyp` (`runtime/cpp/steady_main.cpp:15-26`). Recommendation: do not mark reset complete until the C++ session object includes generation suppression, finalize fork isolation, emitted/committed text, raw/pending/post-stop audio, and a two-utterance/reset/resume trace suite.

16. [MAJOR] The runtime still has per-symbol host syncs in the C++ decode, so the "no per-frame `.item()`" density premise is not implemented. The plan's corrected thesis says the native runtime keeps decode on-GPU with no per-frame `.item()` (`PLAN.md:68-72`). The current C++ decode calls `argmax().item<int64_t>()` inside the symbol loop (`runtime/cpp/steady_main.cpp:46-55`, `runtime/cpp/decode_main.cpp:18-28`). The 0.6a doc also says the on-GPU `.item()`-free formulation remains separate work (`runtime/0.6a-decode-findings.md:55-58`). Recommendation: split "semantic decode port" from "performance decode port"; do not use semantic C++ decode as evidence for the density target until sync-compressed/on-GPU decode exists and is measured.

17. [MAJOR] The production-scope language is still easy to misread as drop-in despite v1 being EN-only. The plan explicitly limits v1 to the English 0.6b checkpoint and drops "drop-in" for prompted/multilingual models (`PLAN.md:319-322`); the final decision repeats EN-only and defers multilingual/prompted (`reviews/0.4-decision-FINAL.md:15-17`). Recommendation: every top-line "production runtime" or "drop-in" statement should carry "EN-only, no prompted/multilingual, no alignments/confidence" until those paths are implemented and tested.

18. [MINOR] The metadata assertions are too shallow to prevent silent artifact/model drift. `steady_main.cpp` asserts only shift, pre, drop, blank, and max_symbols (`runtime/cpp/steady_main.cpp:72-80`). It does not assert model ID, tokenizer hash, att-context, NeMo/torch version, dtype/TF32 settings, encoder artifact kind, cache tensor shapes, or first/steady/finalize geometry identity. Recommendation: bundle and assert all artifact contract metadata at startup.

19. [MINOR] The phase table has duplicated `1.3` rows and overstates completed status relative to the remaining blockers. `PHASE1-STEPS.md` lists two `1.3` rows (`runtime/PHASE1-STEPS.md:24-25`) and marks T1 hardening/T2a done while the T2a wire and finalize/state machine remain pending (`runtime/PHASE1-STEPS.md:21-24`). Recommendation: split "demo/token-exact steady path" from "server-equivalent streaming path" in the status table so downstream readers do not confuse no-crash steady tests with production semantic parity.

20. [MINOR] The plan's "byte-exactness is free with libtorch" language is contradicted by the trace drift found later. The plan says same ATen/cuBLAS/cuDNN makes byte-exact "by construction" (`PLAN.md:301-309`). Later evidence found TorchScript trace drifted about 1e-5 across cache length and required `torch.export` to fix it (`runtime/T2a-findings.md:8-13`). Recommendation: replace "free/by construction" with "possible only under matched versions, exported graph form, geometry coverage, determinism settings, and C++ integration."

## Top 5

1. The density gate is the biggest overclaim: L40S 2-2.5x is measured for encoder replay plus host-sleep decode, not the real decode/finalize dependency chain.
2. The GO decision uses 5090 synthetic finalize sensitivity to clear a deploy-target L40S gate that explicitly has no finalize path.
3. "Byte-exact streaming" is too broad: C++ covers one bundled decode fixture, streaming uses one audio clip, and preprocessing is full-clip-mel slicing rather than incremental STFT.
4. T2a is not in the C++ runtime; `.pt2` export success is not the same as AOTI-compiled C++ byte-exact execution.
5. The current C++ state/reset path is a steady-path demo, not the full multi-utterance server state machine with finalize fork isolation and generation suppression.

Overall verdict: the work is a promising narrow prototype, but the current claims are materially overbanked; the evidence does not yet support production density or byte-exact multi-stream runtime confidence.
