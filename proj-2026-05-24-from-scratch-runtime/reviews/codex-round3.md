# Codex adversarial review - Round 3

Review target: `proj-2026-05-24-from-scratch-runtime/PLAN.md` v3. New findings only, stressing v3 edits.

1. [BLOCKER] v3's 0.6 gate still conflates deployed eager label-looping with an unspecced CUDA-graph decoder rewrite.

   The deployed batch path is explicitly `greedy_batch` with `loop_labels=True` and `use_cuda_graph_decoder=False` (`src/nemotron_speech/server.py:1463-1474`), and the server disables batching if a CUDA graph decoder is detected (`src/nemotron_speech/server.py:976-999`). v3 nevertheless puts "solve the Blackwell cuda-graph-decoder problem" inside the same 0.6 equivalence gate as the deployed decode (`proj-2026-05-24-from-scratch-runtime/PLAN.md:257-265`). That is not a complete spec. NeMo's label-looping computer has three graph modes: full conditional-node graphs, partial graphs with Python while loops, and no graphs (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/transducer_decoding/label_looping_base.py:73-77`). The partial mode still performs host `.item()` loop tests (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/transducer_decoding/rnnt_label_looping.py:658-666`), while the full mode depends on CUDA conditional nodes and custom kernels (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/transducer_decoding/rnnt_label_looping.py:899-930`).

   Recommended change: split 0.6 into `0.6a` deployed eager label-looping equivalence (`use_cuda_graph_decoder=False`, no graph claims) and `3.2/0.6b` graph-decode research. The graph-decode gate must name the required graph mode, conditional-node support, capture/replay ownership, fallback behavior, and whether partial-graph `.item()` loops are allowed. Do not use the graph work as a funding gate for deployed semantic equivalence.

2. [BLOCKER] The alignments/confidence part of 0.6 is testing a mode production does not run, and the fallback mode is different decode code.

   v3 says 0.6 compares alignments/confidence "when enabled" (`proj-2026-05-24-from-scratch-runtime/PLAN.md:262-264`), citing the right server block, but the deployed launcher does not set `NEMOTRON_EOU_PROBE` (`deploy/launch_multiproc.sh:42-45`). The only server path that turns on `preserve_alignments` and `preserve_frame_confidence` is gated by `self.eou_probe_enabled` (`src/nemotron_speech/server.py:1476-1484`, `src/nemotron_speech/server.py:1495-1503`), and that same flag disables batching before decoding is configured (`src/nemotron_speech/server.py:619`, `src/nemotron_speech/server.py:633-634`). In other words, "greedy_batch label-looping with preserve_*" is not the deployed configuration; EOU-probe preservation falls back to the non-batched `greedy` branch unless v3 intentionally changes server behavior.

   There is also a graph-mode trap if v3 keeps graph decoder and preserve_* in the same bucket: eager label-looping adds prior decoded length to alignment timestamps (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/transducer_decoding/rnnt_label_looping.py:486-490`), while the CUDA-graph helper's comment says "add" but the alignment code subtracts (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/transducer_decoding/rnnt_label_looping.py:1162-1175`). v3 does not define which behavior is the oracle.

   Recommended change: make the deployed 0.6 gate explicitly "no alignments/confidence, `greedy_batch`, `use_cuda_graph_decoder=False`." Add a separate EOU-probe gate only if native v1 will support EOU telemetry; that gate must cover the actual fallback decode mode and the emitted EOU payload fields (`src/nemotron_speech/server.py:4039-4069`), or state EOU-probe is out of v1.

3. [BLOCKER] "Mixed-continuation batch rows" in 0.6 contradicts the current server's batching contract.

   v3 requires 0.6 fixtures for "fresh-vs-established rows" and "mixed-continuation batch rows" (`proj-2026-05-24-from-scratch-runtime/PLAN.md:259-262`). NeMo's low-level label-looping state can merge `None` rows by manufacturing a start state (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/transducer_decoding/rnnt_label_looping.py:599-607`), but the deployed server deliberately prevents those mixed batches. The steady batch key includes `session.previous_hypotheses is None` and `session.pred_out_stream is None` (`src/nemotron_speech/server.py:4789-4812`); finalize has equivalent fresh-state keying (`src/nemotron_speech/server.py:7225-7246`); and `stack_hypotheses`/`stack_pred_out` assert uniform None-or-not before a batched call (`src/nemotron_speech/batch_primitives.py:100-139`).

   Recommended change: remove mixed-continuation batches from the deployed equivalence gate, or mark them as a new batching semantics change. If native intends to allow them for higher fill-B, add a separate T1/T2 compatibility gate that proves per-session event streams remain identical and update the trace simulator to model the new grouping rule.

4. [MAJOR] 0.6 still leaves optional n-gram LM fusion and RNNT-vs-TDT `time_jump` semantics unclosed.

   The server-created deployed config does not include an n-gram LM (`src/nemotron_speech/server.py:1468-1474`), but NeMo's `greedy_batch` label-looping class explicitly accepts `ngram_lm_model` and wires it into `NGramGPULanguageModel` (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/rnnt_greedy_decoding.py:647-653`). The label-looping state then carries `lm_states` (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/transducer_decoding/label_looping_base.py:39-48`), combines RNNT logits with LM scores (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/transducer_decoding/rnnt_label_looping.py:345-354`), and updates LM state on chosen labels (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/transducer_decoding/rnnt_label_looping.py:457-464`). v3 says compare label/predictor state and align/conf, but not whether `lm_state` is supported, rejected, or part of the fixture matrix (`proj-2026-05-24-from-scratch-runtime/PLAN.md:257-264`).

   Similarly, v3 lists `time-jump` as part of `LabelLoopingStateItem` (`proj-2026-05-24-from-scratch-runtime/PLAN.md:110-111`), but the RNNT implementation always returns `time_jumps=None` (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/transducer_decoding/rnnt_label_looping.py:494-509`, `/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/transducer_decoding/rnnt_label_looping.py:608-619`). Treating it as a real RNNT behavior is scope creep from TDT-style label looping.

   Recommended change: state that native v1 rejects any non-null `ngram_lm_model` and asserts RNNT `time_jump is None`, or add LM and TDT-style fixtures with exact `lm_state`/`time_jump` equivalence. Do not leave those optional surfaces implicit inside "exact state equivalence."

5. [BLOCKER] Track A is not actually baseline-independent; byte-exact gates are now a moving target because the Python plan intentionally changes the model path.

   v3 calls Track A "baseline-independent" feasibility (`proj-2026-05-24-from-scratch-runtime/PLAN.md:87-96`), but 0.2 and 0.8 are byte-exact-vs-Python gates (`proj-2026-05-24-from-scratch-runtime/PLAN.md:250-270`). The near-term Python plan is explicitly changing finalize padding/graph behavior and relaxing tensor equality because physical T changes can alter numerical paths (`proj-2026-05-24-0859/PLAN.md:62-67`, `proj-2026-05-24-0859/PLAN.md:97-122`). The source supports why this is not a paperwork issue: the current finalize graph key uses exact physical `time_steps` (`src/nemotron_speech/server.py:2154-2160`), the current graph static length is initialized to the bucket's full T (`src/nemotron_speech/cudagraph_encoder.py:193-204`), and the Conformer builds cache/masks from physical `audio_signal.size(1)` and lengths (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/modules/conformer_encoder.py:645-667`).

   v3 also hardens the native plan to "byte-identical encoder output for the REAL frames" under finalize padding (`proj-2026-05-24-from-scratch-runtime/PLAN.md:227-228`, `proj-2026-05-24-from-scratch-runtime/PLAN.md:336-338`), which directly conflicts with the Python plan's relaxed `allclose` tensor gate for padded-T (`proj-2026-05-24-0859/PLAN.md:62-67`). That makes "byte-exact vs Python" a moving and possibly impossible target for Track A.

   Recommended change: define a frozen oracle for Track A: either "same-shape eager NeMo at commit X" or "post-Python production commit Y." If Track A runs before the Python plan lands, require rerunning 0.2/0.8 against the final Python commit before 0.4. Align the padded-T gate with the Python plan: tokens/event stream byte-exact, real-frame tensors allclose unless the source path is exact-shape.

6. [BLOCKER] Native CUDA graph capture/replay is underspecified for a multi-threaded shared-weight runtime.

   v3 adds native CUDA graphs in Phase 3 (`proj-2026-05-24-from-scratch-runtime/PLAN.md:336-340`) while also banking a single shared weight copy across worker threads (`proj-2026-05-24-from-scratch-runtime/PLAN.md:318-319`). The current source avoids that exact concurrency problem by capturing through a single-thread graph executor for the default model (`src/nemotron_speech/server.py:1882-1892`) and separately capturing lane managers on each lane's single-worker executor/stream (`src/nemotron_speech/server.py:1915-1942`). It even logs when a graph is replayed from a different thread than the capture thread on non-lane paths (`src/nemotron_speech/server.py:2960-2971`, `src/nemotron_speech/server.py:2994-3004`).

   The graph object is not stateless. The encoder graph owns static input/output buffers (`src/nemotron_speech/cudagraph_encoder.py:193-204`), replay copies into those buffers and returns the static output pool (`src/nemotron_speech/cudagraph_encoder.py:272-281`), and the module doc warns callers must clone results before another replay (`src/nemotron_speech/cudagraph_encoder.py:15-17`). NeMo's label-looping graph compiler also creates a dedicated capture stream and uses `capture_error_mode="thread_local"` (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/transducer_decoding/rnnt_label_looping.py:839-879`, `/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/transducer_decoding/rnnt_label_looping.py:881-890`).

   Recommended change: before Phase 3, add a CUDA graph ownership contract: one graph/static-buffer pool per lane stream, or a replay mutex with measured lost overlap; capture thread/stream affinity; allocator warm-up and "no allocation during capture" rules; graph memory-pool accounting; and explicit clone/lifetime rules for outputs. Without that, shared weights plus native graph replay can silently serialize or corrupt outputs.

7. [MAJOR] Exact-B graph buckets and continuous batching are still treated as compatible without a hit-rate/memory proof.

   v3 says continuous batching should fill B in Phase 2 and then native steady graphs cover `B=1..K` in Phase 3 (`proj-2026-05-24-from-scratch-runtime/PLAN.md:327-340`). The current manager captures exact batch sizes and does not pad smaller batches into larger buckets (`src/nemotron_speech/cudagraph_encoder.py:6-8`); replay requires exact input shapes (`src/nemotron_speech/cudagraph_encoder.py:263-281`, `src/nemotron_speech/cudagraph_encoder.py:578-599`); and the server returns no graph bucket if B is outside the captured range or uncaptured (`src/nemotron_speech/server.py:2080-2084`). That means a native runtime with B varying from 1..K needs a graph and static pool per exact B per lane, or it falls back to eager for the long tail of B values.

   This matters because the deployed memory cap is already graph-pool-bound: L40S is K=3 because K=4 OOMs with model + lane + finalize graph pools (`deploy/launch_multiproc.sh:6-9`, `deploy/launch_multiproc.sh:36-45`). v3's 0.5 simulator reports achievable B distribution, but not exact-B graph hit rate, graph-pool memory, capture time, or eager fallback cost (`proj-2026-05-24-from-scratch-runtime/PLAN.md:295-299`).

   Recommended change: extend 0.5/3.1 with a graph-bucket capacity model: per-lane graph memory for each B, expected replay hit rate from trace-derived B histograms, fallback percentage, and a cap policy for rare B values. Do not count graph speedups or shared-weight density until that model passes on L4 and L40S memory limits.

8. [MAJOR] The determinism gate ignores cross-batch-size and shared-weight numerical drift.

   v3 makes T2 byte-exact for B1a across B>1 geometries (`proj-2026-05-24-from-scratch-runtime/PLAN.md:205-210`, `proj-2026-05-24-from-scratch-runtime/PLAN.md:251-256`), but it does not define whether a session must be identical when decoded alone versus batched with other rows. The source changes tensor shapes when batching: mels are concatenated to `[B,F,T]` (`src/nemotron_speech/batch_primitives.py:59-75`), caches are concatenated inside the layer dimension (`src/nemotron_speech/batch_primitives.py:78-87`), and the Conformer then runs all layers over the batched tensors (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/modules/conformer_encoder.py:677-695`). That can change GEMM/reduction shape even if every per-session input is identical.

   The current server also only disables TF32 when batching is enabled (`src/nemotron_speech/server.py:655-660`). A native runtime sharing one weight copy across lanes must pin TF32, cuBLAS/cuDNN algorithm choices, and workspace behavior explicitly; otherwise "same weights" does not imply byte-identical outputs across streams, lanes, or batch sizes.

   Recommended change: split determinism into two gates: same-shape native-vs-Python byte-exactness, and cross-B/session-invariance with a named tolerance plus exact token/event equivalence. Add a startup/runtime contract for TF32, cuBLAS workspace config, deterministic algorithm flags, and per-lane handle/workspace ownership.

9. [MAJOR] The v3 effort estimate is still fantasy, now with false precision.

   v3 estimates native label-looping equivalence at 3-5 engineer-weeks and the whole production runtime at 22-35 engineer-weeks (`proj-2026-05-24-from-scratch-runtime/PLAN.md:386-408`). Packing/unpacking `Hypothesis` and alignment data moves data to CPU and rebuilds Python objects (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/utils/rnnt_utils.py:737-808`), so a native equivalent is not just kernels.

   Outside decode, v3 includes a drop-in WebSocket server and protocol (`src/nemotron_speech/server.py:4148-4248`), bounded queues/backpressure (`src/nemotron_speech/server.py:4326-4341`), scheduler and finalize metrics (`src/nemotron_speech/server.py:5388-5424`, `src/nemotron_speech/server.py:7254-7281`), CUDA graph pools (`src/nemotron_speech/cudagraph_encoder.py:284-296`), multi-process/MPS deploy hardening with known TODOs (`deploy/launch_multiproc.sh:70-79`), and multi-platform validation. The production budget should include at least separate line items for native decode, CUDA graph infrastructure, scheduler/WS/protocol, observability, deployment/drain/rollback, multi-platform CI/sweeps, and long-tail correctness fuzzing. Add staffing/calendar assumptions, not only "1 senior eng +-50%."

## Top 5 things to fix

1. Split 0.6 into deployed eager label-looping equivalence and later graph-decode research; do not mix Blackwell CUDA graph work into the deployed semantic gate.
2. Freeze the Track A oracle or rerun Track A after the Python plan lands; align padded-T equality with the Python plan's relaxed tensor gate.
3. Add a CUDA graph ownership contract for native multi-threading: per-lane graph pools or a measured replay mutex, capture thread/stream rules, allocator rules, and output lifetime rules.
4. Remove or reclassify mixed-continuation batching, LM fusion, EOU preserve_* telemetry, and `time_jump` unless v1 explicitly supports them.
5. Replace the 22-35 engineer-week estimate with a prototype-vs-production budget that accounts for native decode, graph infrastructure, WS/scheduler, observability, deploy, and multi-platform validation.
