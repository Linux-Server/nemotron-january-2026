## Pinned decisions
- **Language / shape: ALL-C++ model+serving core** (shape 1). Rationale: the entire hot path is libtorch + CUDA-graph +
  c10::cuda streams/allocator + the native decode — all C++-first. **tch-rs CUDA-graph capture is UNIMPLEMENTED
  (tch-rs#631)** → all-Rust is vetoed for the hot path; a Rust front would add an FFI seam for no benefit on the
  concurrency that matters. (Revisit a Rust networking front only if a future need justifies the seam.)
- **Version pin: libtorch 2.8.0+cu128 + NeMo 2.4.1** (the proven-working pair; EN ckpt cached; sm_120 in arch_list).
  SAME version for export + golden fixtures + runtime (T2a). C++ ABI = cxx11 (`_GLIBCXX_USE_CXX11_ABI=1`).
  **Build:** manual-link libtorch (no nvcc) works for graph-replay; **Wave-2 custom kernels need a glibc-compatible CUDA
  toolchain** (CUDA devel container or an Ubuntu-22.04/24.04 build env — the local box's glibc 2.41 breaks nvcc).
- **v1 scope: English 0.6b only**, deployed config: `greedy_batch` label-looping, `use_cuda_graph_decoder=False`,
  `eou_probe_enabled=False` (no alignments/confidence), no n-gram LM, RNNT `time_jump` asserted None. Multilingual/
  prompted = later phase.
- **Correctness bar (T1, the SHIP gate):** full-1000 WER **non-inferior to the Python baseline within the established CI**
  (baseline 1.95% — memory `silence0-warm200-shippable`; treat ≈±0.2–0.3% abs as the band, finalize the exact width
  against a fresh baseline run) + **per-utterance WER bounded** + **exact per-session event-stream / final-delta /
  generation-suppression equivalence**. **T2a:** B1a encoder byte-exact vs frozen same-shape fixtures on one arch.
- **Fleet:** L40S/Ada-density (L4 BW-bound, out for the native density case). **Cost basis: strategic capability bet**
  (no COGS break-even gate; the ≥1.5× floor must clear — it did).

## What GO funds (Phase 1), and the residual risk
**GO funds the native C++ runtime build**, BUT the dominant remaining risk is **0.6a — native `greedy_batch`
label-looping decode, byte/state-exact vs NeMo** (the un-exportable, research-grade part). 0.6a is therefore **Phase 1's
first step and a hard sub-gate**: if it can't reach exact `Hypothesis`/`LabelLoopingStateItem` equivalence, B1 either
accepts a named T1-only-decode risk or STOPs (no B4/B5 fallback). Everything downstream (steady path, scheduler,
batching) is comparatively standard engineering.

## Quantified expectation (honest)
Native density on L40S ≈ **2× the production ~16–20/box** (→ ~32–40/box, GPU-bound by the real encoder), in ONE process
(no MPS tax, shared weights). Zero p50 movement (VAD+WAN-bound). The win is **density + tail**, as scoped.

**DECISION: GO. Proceed to Phase 1, starting with 0.6a (decode equivalence) as the build's first hard sub-gate.**
