   Evidence: C++ accepts non-byte-equal mel whenever `max_abs <= 5.0e-4` (`/home/khkramer/src/nemotron-january-2026/proj-2026-05-24-from-scratch-runtime/runtime/cpp/session_main.cpp:1430`, `/home/khkramer/src/nemotron-january-2026/proj-2026-05-24-from-scratch-runtime/runtime/cpp/session_main.cpp:1438`, `/home/khkramer/src/nemotron-january-2026/proj-2026-05-24-from-scratch-runtime/runtime/cpp/session_main.cpp:1451`).    Why I care: `byte_equal=17/1204` with `max_abs_diff=2.44e-4` is plausibly benign cuFFT/libtorch/Python-process drift, especially because chunk count, geometry, tokens, events, raw ring, and counters pass. But the code currently cannot explain why `5e-4` is the right line instead of `3e-4` or `1e-3`. That makes the result easy to overclaim.

   Fix: turn `mel_atol` into a named, versioned preproc CI bound. Update `validate_preproc.py` or add a sibling gate that runs the exported `preproc.ts` over the bundle PCM clips with the same fixed `K`, reports max/mean/p99 abs diff and relative diff, and writes the accepted threshold rationale. Keep token/event exact as the semantic bar, but make mel acceptance "within documented preproc CI", not "within whatever constant is in C++".

Step 3's audio-front equivalence is met at the current semantic gate: PCM-fed C++ produces the same steady/final token stream, the same event text stream, the same fork-parent preservation, exact recomputed finalize geometry, exact raw-ring/counter state, and no mel diff above the current tolerance.

The `2.44e-4` mel non-byte-equality is acceptable as a Step-3 result only if it is described as bounded preprocessor numerical drift under a token/event/WER gate, not as byte-equivalent preprocessing. The implementation evidence distinguishes it from an obvious alignment bug because chunk boundaries, valid-frame slices, raw-ring advancement, retained audio state, and final geometry all match the Python reference. The remaining risk is not an observed bug; it is an under-specified CI contract. Fix the tolerance provenance, margin reporting, and `preproc.ts` provenance before calling this robust beyond the 20/20 audio bundle.
