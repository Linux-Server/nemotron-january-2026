# Codex adversarial review - Round 2

Review target: `proj-2026-05-24-from-scratch-runtime/PLAN.md` v2. This does not restate round-1 findings unless the v2 fix is still wrong.

1. [BLOCKER] v2 targets the wrong RNNT decode path for the deployed server.

   The plan now says the current decode is "NeMo greedy" with `loop_labels=False => FRAME-looping` and makes "native RNNT frame-looping decode" the go/no-go (`PLAN.md:92-98`, `PLAN.md:234-237`). That is not the deployed path. The production launcher enables `NEMOTRON_BATCH_SCHED=1` and `NEMOTRON_BATCH_FINALIZE=1` (`deploy/launch_multiproc.sh:42-45`). In that mode, the server forcibly sets `decoder_strategy = "greedy_batch"` when batching is enabled (`src/nemotron_speech/server.py:726-728`), and `load_model()` configures `strategy='greedy_batch'`, `max_symbols=10`, `loop_labels=True`, `use_cuda_graph_decoder=False` (`src/nemotron_speech/server.py:1463-1474`). `loop_labels=False` appears only in the non-batched `strategy='greedy'` branch (`src/nemotron_speech/server.py:1486-1492`).

   NeMo confirms the distinction: `strategy == 'greedy'` instantiates `GreedyRNNTInfer` and does not use the `loop_labels` switch (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/rnnt_decoding.py:305-319`); `strategy == 'greedy_batch'` instantiates `GreedyBatchedRNNTInfer` and passes `loop_labels` plus `use_cuda_graph_decoder` (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/rnnt_decoding.py:350-365`). In `GreedyBatchedRNNTInfer`, `loop_labels=True` selects the label-looping computer (`rnnt_greedy_decoding.py:635-647`), while frame-looping is only the `loop_labels=False` branch (`rnnt_greedy_decoding.py:654-686`).

   Recommended plan change: replace "native RNNT frame-looping decode" with "native implementation of the exact live decoder strategy." For the deployed/batched runtime that means `greedy_batch` + `loop_labels=True` label-looping, including its state object. Keep a separate frame-looping sub-spike only if the plan explicitly disables batching or deliberately changes the Python baseline and re-runs all throughput claims.

2. [BLOCKER] Spike 0.6 is still underspecified even after correcting the decoder target.

   The hard part is not just "blank handling, max_symbols, partial-hypothesis continuation" in text. The live label-looping path carries a `LabelLoopingStateItem` with predictor state/output, label, decoded length, optional LM state, and time jump (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/transducer_decoding/label_looping_base.py:38-59`). Continuation merges partial hypotheses in place for label-looping (`rnnt_greedy_decoding.py:783-804`). By contrast, NeMo's batched frame-looping implementation explicitly rejects `partial_hypotheses` (`rnnt_greedy_decoding.py:807-815`), so using it as the native model would break streaming continuation.

   The native spike also has to match label-looping's non-obvious semantics: forced blank/time advance after `max_symbols` (`rnnt_label_looping.py:466-484`), timestamp and decoded-length carry across iterative chunks (`rnnt_label_looping.py:486-506`), split/merge of batched state for per-session carry (`rnnt_label_looping.py:569-620`), and optional CUDA-graph modes that depend on conditional-node support (`label_looping_base.py:96-115`). NeMo then packs hypotheses by converting `y_sequence` and moving decoder state to CPU (`rnnt_greedy_decoding.py:52-86`), while `Hypothesis.merge_()` mutates score, sequence, state, timestamps, alignments, confidence, and invalidates text (`rnnt_utils.py:146-173`). The repo's fork code knows this is alias-sensitive: clone helpers recurse through NeMo dataclasses/objects (`src/nemotron_speech/server.py:147-207`, `src/nemotron_speech/server.py:249-263`), finalization forks deep-clone hypothesis and prediction state (`src/nemotron_speech/server.py:6370-6425`), and batching rejects hypothesis aliasing (`src/nemotron_speech/batch_primitives.py:109-126`).

   Recommended plan change: define 0.6 outputs as exact `Hypothesis`/state equivalence, not just text or WER on fixed encoder outputs. Fixtures must serialize previous hypotheses, label-looping state items, forked finalization state, fresh-vs-established rows, `max_symbols` saturation cases, all-blank chunks, and batch rows with mixed continuation state. Compare `y_sequence`, score, timestamp/length, `last_token`/label state, decoder-state shapes/devices/dtypes, alignments/confidence when enabled, and parent-fork unchanged assertions.

3. [BLOCKER] The "worth-it gate" sequencing is internally inconsistent and makes 0.1/0.5 non-falsifiable.

   Section 0 says the worth-it gate is "HARD - precedes all spikes" (`PLAN.md:68`), but the gate then says Phase-0 spikes 0.1-0.5 may run in parallel with the Python plan (`PLAN.md:75-80`). That is only safe for narrow feasibility spikes. It is not coherent for 0.1 and 0.5 as written because both are supposed to prove they close a named residual gap after the Python plan lands.

   The current scheduler behavior that 0.5 must simulate is tightly tied to live settings: queueing uses bounded scheduler queues and `await queue.put()` (`src/nemotron_speech/server.py:4326-4341`), batching uses `NEMOTRON_BATCH_MAX_WAIT_MS`/`MAX_SIZE` (`src/nemotron_speech/server.py:665-670`), group keys include language, `keep_all_outputs`, `drop_extra`, chunk T, decoder mode, and fresh/established decoder-state flags (`src/nemotron_speech/server.py:4789-4812`), and dispatch selection is deadline/size/lane-affinity dependent (`src/nemotron_speech/server.py:4864-4918`). The Python plan is explicitly changing admission, maxconn, padded finalize graphs, host syncs, finalize priority, and the GIL/decode probe before measuring the residual gap (`proj-2026-05-24-0859/PLAN.md:83-145`). Pre-plan traces will not answer whether a post-plan native runtime is worth funding.

   Recommended plan change: split Phase 0 into "parallel feasibility" and "post-Python residual validation." 0.2/0.7 and a correctly targeted decoder micro-spike can run early. 0.1, 0.3, and 0.5 should not produce go/no-go claims until the Python plan has landed, the exact baseline commit is recorded, and post-plan traces/telemetry are captured. `reviews/decision.md` should name which measurements came from pre-plan feasibility versus post-plan residual-gap validation.

4. [MAJOR] Spike 0.1 does not isolate finalize/steady overlap from existing gates, batch-finalize behavior, lane affinity, and MPS.

   v2 says 0.1 will "drive the real finalize + steady scheduler path" on a second dispatcher thread "without the exclusive-gate stall" (`PLAN.md:214-220`). The current deployed path is already not a simple exclusive-gate path: with `NEMOTRON_BATCH_FINALIZE=1` (`deploy/launch_multiproc.sh:42-45`), finalize uses `_scheduler_pinned_model_lane_path()` and directly calls the lane worker (`src/nemotron_speech/server.py:6755-6773`). Only the fallback path takes `inference_lock` (`src/nemotron_speech/server.py:6779-6785`), and only non-parallel chunks use the coarse exclusive path (`src/nemotron_speech/server.py:3213-3233`, `src/nemotron_speech/server.py:5310-5323`).

   The missing confound is lane affinity. A finalize can overlap other work only if the relevant lane is free; the pinned-lane path waits while its lane is unavailable or the same session is in flight (`src/nemotron_speech/server.py:6711-6720`), and sessions are pinned by `_scheduler_assign_session_model_lane()` (`src/nemotron_speech/server.py:3295-3308`). Each lane worker still ends with `stream.synchronize()` (`src/nemotron_speech/server.py:3175-3178`). The deploy shape also starts CUDA MPS and multiple processes (`deploy/launch_multiproc.sh:57-68`), so "second dispatcher thread" results can be dominated by process/context/MPS differences rather than the runtime design.

   Recommended plan change: make 0.1 an ablation matrix, not a single comparison. Hold the model/decode constant and separately toggle: batch-finalize on/off, exclusive gate on/off, `inference_lock` on/off, lane-end sync versus CUDA event, same-lane versus cross-lane finalize/steady, single-process/single-context versus MPS versus multi-process, and CPU thread affinity. Report per-lane CUDA-event timelines and queue/lane wait, not just throughput knee.

5. [MAJOR] "One shared read-only weight copy" ignores mutable NeMo/module state that lane replicas currently avoid.

   The density thesis assumes native worker threads can share one read-only model. The source has several mutable per-call model surfaces. Prompted models call `set_inference_prompt()` on the current inference model before a batch/chunk (`src/nemotron_speech/server.py:1288-1292`, `src/nemotron_speech/server.py:8306-8307`, `src/nemotron_speech/server.py:7394-7395`). The encoder wrapper mutates `model.encoder.streaming_cfg.drop_extra_pre_encoded` and restores it afterward (`src/nemotron_speech/cudagraph_encoder.py:56-64`; also `src/nemotron_speech/batch_primitives.py:142-149`). NeMo greedy forwards temporarily switch decoder/joint modules to eval and then restore training state (`rnnt_greedy_decoding.py:390-415`, `rnnt_greedy_decoding.py:744-766`). Today, per-lane replicas are created with separate NeMo model objects (`src/nemotron_speech/server.py:3109-3137`), which sidesteps these shared-object races at a memory cost.

   Recommended plan change: add a Phase-0/1 "model mutability audit" before claiming shared weights. The design must separate immutable parameter storage from mutable per-request context: prompt/language state, streaming/drop-extra config, decoder temporary state, graph static buffers, RNG/training flags, and any NeMo module fields touched during inference. If the native path keeps a shared module object, TSan will not prove logical prompt/drop-extra isolation.

6. [MAJOR] T1 still misses prompted/multilingual protocol equivalence.

   T1 now covers event streams, deltas, WER, and stale suppression (`PLAN.md:165-174`), but not the language/prompt semantics that are already in the server. Connection validation rejects unsupported `language` for non-prompted models and unsupported languages for prompted models (`src/nemotron_speech/server.py:1350-1377`). Prompted inference mutates model prompt state per session (`src/nemotron_speech/server.py:1288-1292`). Text extraction strips complete and partial language tags from cumulative hypotheses (`src/nemotron_speech/server.py:1379-1397`). Batch grouping depends on target language for steady and finalize rows (`src/nemotron_speech/server.py:4796-4803`, `src/nemotron_speech/server.py:7225-7246`). NeMo tokenizer decoding can also expose language IDs for aggregate tokenizers (`rnnt_decoding.py:1803-1854`, `rnnt_decoding.py:1856-1889`).

   Recommended plan change: make prompted/multilingual a required T1 axis, not "if enabled" only in 0.2. Add simultaneous multi-language sessions, unsupported query rejection, prompt-tag stripping, partial language-tag boundaries, and same-batch language isolation tests. If v1 is English-only, say so as an explicit product-scope reduction and remove "drop-in" for prompted models.

7. [MAJOR] T1's reset/finalize/resume coverage is still too shallow for the continuous state machine.

   Plan 1.3 mentions "resume-after-stop + stale-generation suppression" (`PLAN.md:252-254`), but the source has more interleavings than that. `vad_stop` invalidates generation, arms debounce, and enters `PENDING_FINALIZE` (`src/nemotron_speech/server.py:5755-5770`). Audio arriving while pending finalize is held in `continuous_post_stop_audio` instead of appended (`src/nemotron_speech/server.py:5590-5598`). `vad_start` cancels debounce and flushes that held audio back into the stream (`src/nemotron_speech/server.py:5731-5744`). `reset` while pending does not finalize immediately; it sets `continuous_reset_seen` and waits for debounce (`src/nemotron_speech/server.py:5823-5829`). True-boundary `end`/`close` force finalize and either include or discard post-stop audio (`src/nemotron_speech/server.py:5676-5684`, `src/nemotron_speech/server.py:5802-5815`). Finalize checks generation before model work and again before emit (`src/nemotron_speech/server.py:7301-7310`, `src/nemotron_speech/server.py:7660-7679`, `src/nemotron_speech/server.py:7795-7813`, `src/nemotron_speech/server.py:6827-6842`), then may cold-reset and invalidate again (`src/nemotron_speech/server.py:5907-5928`).

   Recommended plan change: T1 needs a state-machine trace suite with exact event order and emitted JSON for: audio-after-vad_stop, vad_start-before-debounce, reset-before-debounce, end/close while pending, reset during in-flight finalize, stale finalize after newer audio, and close while send is tolerated. Compare generation transitions and post-stop audio byte counts, not just final transcript.

8. [MAJOR] The project is still not bounded as an engineering plan; operational/runtime parity is deferred into two vague Phase-5 bullets.

   v2 lists a Rust/C++ serving core, native RNNT decode, CUDA graphs, continuous batching, websocket server, deploy integration, and rollout (`PLAN.md:245-283`) with no team size, calendar, staffing assumptions, or kill criteria beyond individual spikes. The missing work is not optional glue. The current server defines connection validation, ready/error messages, cleanup, and close behavior in the websocket handler (`src/nemotron_speech/server.py:4148-4248`), uses bounded scheduler queues for backpressure (`src/nemotron_speech/server.py:4326-4341`), emits interim/final JSON with specific payload fields (`src/nemotron_speech/server.py:5234-5244`, `src/nemotron_speech/server.py:6849-6857`), has batch/finalize telemetry (`src/nemotron_speech/server.py:5388-5424`, `src/nemotron_speech/server.py:7254-7281`), and exposes only a minimal health endpoint (`src/nemotron_speech/server.py:8842-8847`). The deploy launcher explicitly has TODOs for LB drain, alerting, and MPS-context restart after crashes (`deploy/launch_multiproc.sh:70-79`).

   Recommended plan change: before Phase 1, add a runtime contract document and acceptance tests for wire protocol, error payloads, close/reconnect behavior, health/readiness, metrics/log schema, admission/backpressure signals, graceful drain/restart, model load failure, OOM/fallback behavior, and rollback triggers. Add an explicit estimate: people, calendar, platform bring-up time, and what is out of v1. Without that, this is a research program, not a bounded runtime project.

9. [MINOR] The B1a export gate weakens its own byte-exact requirement.

   The correctness section says T2 byte-exact encoder/cache output is a gate for B1a on a single 5090 (`PLAN.md:172-174`), but 0.2 says "Go: T2 byte-exact (or documented T1)" (`PLAN.md:221-225`). For B1a, the source path is supposed to call the same encoder function with the same inputs; the current CUDA-graph wrapper calls exactly `model.encoder.cache_aware_stream_step` and restores `drop_extra_pre_encoded` (`src/nemotron_speech/cudagraph_encoder.py:231-241`, `src/nemotron_speech/cudagraph_encoder.py:56-64`). If that is not byte-exact on one machine, it is not a harmless documentation note; it means export changed behavior or the fixture is not controlling inputs.

   Recommended plan change: remove "or documented T1" from the B1a go criterion. If B1a is not T2 byte-exact on 5090 for first/steady/finalize/B>1 geometries, the decision memo must either no-go B1a or explicitly reclassify it as a non-identical backend with B2-level correctness risk.
