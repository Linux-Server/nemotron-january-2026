# Codex adversarial review - Round 1

Review target: `proj-2026-05-24-from-scratch-runtime/PLAN.md`.

1. [BLOCKER] The core thesis over-assigns the launch ceiling to the GIL and under-assigns it to explicit serializers already in the code.

   The plan says the prize is "per-lane dispatcher threads free of the GIL" and treats reliable capacity as unlocked by "no GIL, no MPS tax" (`proj-2026-05-24-from-scratch-runtime/PLAN.md:29-39`). The source does not support that as the primary explanation. The current server already offloads model calls through `run_in_executor` (`src/nemotron_speech/server.py:3087-3095`), and the multi-lane path already creates one single-worker executor per lane plus one CUDA stream per lane (`src/nemotron_speech/server.py:3155-3163`). The worker executes the model call under `torch.cuda.stream(stream)` and then synchronizes that lane stream before returning (`src/nemotron_speech/server.py:3175-3178`). That is not "only the GIL"; it is a deliberately synchronous lane-call boundary.

   The more concrete serializers are visible in source: `inference_lock` is an `asyncio.Lock` (`src/nemotron_speech/server.py:593-594`); single-lane batching holds it across the whole model call (`src/nemotron_speech/server.py:4990-5005`); non-batched multi-lane chunks go through `_scheduler_exclusive_model_path` and then hold `inference_lock` across the lane call (`src/nemotron_speech/server.py:5310-5323`); finalize fallback paths hold it across final model work (`src/nemotron_speech/server.py:6779-6785`, `src/nemotron_speech/server.py:7698-7734`, `src/nemotron_speech/server.py:7738-7771`). The multi-lane logic also forces non-steady work to wait until in-flight lane tasks drain: only "steady normal chunks" may share lanes, while first chunks, finalization, and other geometries use the pinned lane exclusively (`src/nemotron_speech/server.py:3242-3249`), and non-parallel keys dispatch only when no lane task is in flight (`src/nemotron_speech/server.py:3271-3276`). The scheduler itself is a single asyncio loop, and the code comment documents that without an explicit `sleep(0)`, socket I/O can be starved (`src/nemotron_speech/server.py:4456-4491`).

   Recommended change: rewrite the thesis to name the actual serializers: single asyncio scheduler/I/O thread, `asyncio.Lock` scopes, exclusive pinned-lane gating, lane-end stream synchronization, and Python RNNT decode. Spike 0.1 must run the real scheduler/finalize/steady workload, not a generic "N host threads replay a graph" microbench.

2. [BLOCKER] The plan admits single-context launch serialization only as a risk, but still budgets "no MPS tax" as if it were resolved.

   The plan's target table claims capacity comes from "no GIL, no MPS tax, no Kx11 GB graph-pool dup" (`proj-2026-05-24-from-scratch-runtime/PLAN.md:34-39`), while its own risk section admits a single CUDA context may serialize launches and that MPS/green contexts may be required (`proj-2026-05-24-from-scratch-runtime/PLAN.md:252-254`). The current lane design is one process with multiple streams (`src/nemotron_speech/server.py:3155-3163`), not proof that a single CUDA context can feed the GPU enough. The current deploy explicitly uses multiple processes plus MPS, and `auto_pick_K` caps L40S at K=3 because K=4 OOMs with the current graph pools (`deploy/launch_multiproc.sh:6-9`, `deploy/launch_multiproc.sh:19-24`).

   The plan also blurs graph-pool duplication with model duplication. The current multi-lane code loads an additional full NeMo model for each lane beyond lane 0 (`src/nemotron_speech/server.py:3109-3137`) and stores lane models separately (`src/nemotron_speech/server.py:3150-3154`). A native runtime might share read-only weights across dispatcher threads, but that is a different density mechanism from "remove MPS tax" and from "collapse finalize graph buckets."

   Recommended change: make Spike 0.1 explicitly compare single-process/single-context multi-stream, multi-process MPS, and any green-context option on the same replay workload and on the real finalize+steady scheduler path. Do not claim "no MPS tax" or 40-48 streams/box until that comparison says a single native process actually overlaps work.

3. [BLOCKER] B1 is not "medium effort" and "fidelity by construction"; the export target includes an unexportable Python RNNT control loop.

   The plan's B1 row says libtorch + TorchScript/`torch.export` has "highest" fidelity and "medium" effort (`proj-2026-05-24-from-scratch-runtime/PLAN.md:104-110`). The actual server calls `model.conformer_stream_step`, not just an encoder module (`src/nemotron_speech/server.py:8317-8328`, `src/nemotron_speech/server.py:8797-8809`). NeMo's `conformer_stream_step` first runs `self.encoder.cache_aware_stream_step` and then, for RNNT, calls `self.decoding.rnnt_decoder_predictions_tensor(... partial_hypotheses=previous_hypotheses)` (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/mixins/mixins.py:646-660`, `/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/mixins/mixins.py:705-710`). The function signature itself exposes Python object state: `previous_hypotheses: List[Hypothesis]` and `previous_pred_out` (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/mixins/mixins.py:592-604`).

   The default server configuration is also misstated by the plan. The plan calls the current RNNT path "greedy label-looping" while citing `loop_labels=False` (`proj-2026-05-24-from-scratch-runtime/PLAN.md:61-62`). In source, the default greedy config sets `max_symbols=10`, `loop_labels=False`, and `use_cuda_graph_decoder=False` (`src/nemotron_speech/server.py:1486-1492`). NeMo documents `loop_labels=False` as frame-looping, not label-looping (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/rnnt_greedy_decoding.py:588-594`).

   The actual greedy RNNT loop has data-dependent Python control flow and host syncs: it iterates over each batch row (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/rnnt_greedy_decoding.py:401-409`), then over time (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/rnnt_greedy_decoding.py:447-448`), then in a `while not_blank` loop bounded by `max_symbols` (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/rnnt_greedy_decoding.py:453-480`), calls `.item()` on the argmax token (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/rnnt_greedy_decoding.py:478-480`), and mutates hypothesis state (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/rnnt_greedy_decoding.py:492-504`). That is exactly the part the plan wants to move "on-GPU" with "no per-frame `.item()`" (`proj-2026-05-24-from-scratch-runtime/PLAN.md:202-204`, `proj-2026-05-24-from-scratch-runtime/PLAN.md:226-227`), but it is not obtained by exporting NeMo modules to libtorch.

   Recommended change: split B1 into B1a encoder export and B1b native RNNT greedy decode. B1b needs its own Phase-0 spike on fixed encoder-output fixtures, including partial-hypothesis continuation, `max_symbols=10`, blank termination, decoder state carry, and final/fork behavior. Reclassify B1 effort as high until B1b passes.

4. [MAJOR] Encoder export is also less clean than the plan implies because the streaming encoder path mutates module config and depends on cache geometry.

   The existing CUDA graph manager only captures `model.encoder.cache_aware_stream_step`, not RNNT decode (`src/nemotron_speech/cudagraph_encoder.py:1-4`). That encoder path mutates `self.streaming_cfg.drop_extra_pre_encoded` before the call and restores it afterward (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/mixins/streaming.py:41-55`, `/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/mixins/streaming.py:70-75`). The Conformer forward path changes lengths when `drop_extra_pre_encoded > 0` and cache is present (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/modules/conformer_encoder.py:629-640`), constructs masks using cache lengths and offsets (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/modules/conformer_encoder.py:645-667`), loops layer-by-layer through cached Conformer layers (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/modules/conformer_encoder.py:677-695`), and post-processes outputs differently depending on `keep_all_outputs` and streaming config (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/modules/conformer_encoder.py:523-546`). The relative-position attention path includes `rel_shift`, implemented with pad/view/slice shape manipulation (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/multi_head_attention.py:259-270`, `/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/parts/submodules/multi_head_attention.py:313-314`).

   Recommended change: do not phrase Spike 0.2 as "export `encoder.cache_aware_stream_step` and compare one steady chunk + one finalize." The spike must test all live geometry: first chunk `drop_extra=0`, steady `drop_extra=self.drop_extra`, finalize `keep_all_outputs=True`, batch B>1, prompted/non-prompted target language if enabled, and exact cache round-trip shapes.

5. [MAJOR] T1 is too weak for a streaming server and would miss regressions the code is explicitly protecting against.

   The plan demotes byte-exactness to "aspirational" and lets WER-within-CI plus event sequencing ship (`proj-2026-05-24-from-scratch-runtime/PLAN.md:127-140`). That is dangerous because the source has streaming semantics that aggregate WER cannot see. Final emission computes append-only deltas from cumulative text (`src/nemotron_speech/server.py:6816-6825`), suppresses stale finalizations by generation (`src/nemotron_speech/server.py:6827-6842`), updates committed and emitted text before/after sending (`src/nemotron_speech/server.py:6844-6866`), and suppresses empty/duplicate finals (`src/nemotron_speech/server.py:6881-6895`). Interim emissions are similarly generation-gated (`src/nemotron_speech/server.py:5227-5244`). A native engine can keep final WER inside CI while changing interim flicker, partial/final delta boundaries, duplicate suppression, or stale-generation behavior.

   The state being protected is not trivial. `ASRSession` carries encoder cache tensors plus decoder hypothesis state (`src/nemotron_speech/server.py:468-475`), finalize forks deep-clone caches, hypotheses, prediction state, and text state (`src/nemotron_speech/server.py:6370-6425`), and the flush path asserts the parent fork state stayed unchanged (`src/nemotron_speech/server.py:6792-6797`). Batched hypothesis handling is alias-sensitive because NeMo mutates hypothesis objects in place (`src/nemotron_speech/batch_primitives.py:12-14`, `src/nemotron_speech/batch_primitives.py:109-126`).

   Recommended change: make T1 streaming-specific: exact per-session ordered event stream, exact final-delta text, exact duplicate/stale suppression decisions, per-utterance WER bounds, corpus WER CI with a named numeric threshold, and byte-exact encoder/cache outputs on a single 5090 for B1 where the same ATen kernels are claimed.

6. [MAJOR] The 6-10 ms finalize target is copied from an aspirational fused-runtime roofline, but the selected v1 backend explicitly defers the mechanisms needed to reach it.

   The plan says server finalize should drop from ~22 ms to 6-10 ms (`proj-2026-05-24-from-scratch-runtime/PLAN.md:34-39`) and makes Phase 3's gate "finalize latency measured -> 6-10 ms target on L40S" (`proj-2026-05-24-from-scratch-runtime/PLAN.md:222-228`). The roofline says L40S measured 9.7 ms encoder, 13.4 ms model, and 22 ms wall against a 2.9-3.0 ms memory floor (`proj-2026-05-23-1731/roofline-COMBINED.md:13-23`). It also says fp16 was 0.79x slower at this tiny M and only helps once fused (`proj-2026-05-23-1731/roofline-COMBINED.md:20-23`). The from-scratch roofline projection ties 6-10 ms to fused encoder, fp16/fp8 once fused, fixed-trip/graphed decode, CUDA-event dependency tracking, and no per-chunk sync/scatter (`proj-2026-05-23-1731/roofline-COMBINED.md:56-60`).

   The plan's default backend B1 is libtorch export, while TensorRT and hand kernels are deferred (`proj-2026-05-24-from-scratch-runtime/PLAN.md:108-111`). Existing Python already CUDA-graphs the encoder-only finalize replay (`src/nemotron_speech/cudagraph_encoder.py:1-4`, `src/nemotron_speech/server.py:2316-2368`), and the source still has preprocessor looping, model sync, and scatter sync around finalize (`src/nemotron_speech/server.py:6927-6942`, `src/nemotron_speech/server.py:7294-7342`, `src/nemotron_speech/server.py:7416-7432`, `src/nemotron_speech/server.py:7513-7518`). A B1 libtorch port without fusion does not by itself turn a 24-layer small-M memory-bound Conformer into a 6 ms finalize.

   Recommended change: separate targets by backend. B1 should have a conservative "parity or modest sync/scatter reduction" target. 6-10 ms should be explicitly contingent on a later fused encoder/decode milestone with its own proof, not a Phase 3 gate for the default B1 runtime.

7. [MAJOR] The 3-5x throughput target assumes batching will fill B, but the current source shows strict grouping constraints and arrival timing are the hard part.

   The plan says continuous batching will fill B without adding user wait (`proj-2026-05-24-from-scratch-runtime/PLAN.md:214-216`) and claims 3-5x steady throughput from B-fill (`proj-2026-05-24-from-scratch-runtime/PLAN.md:36-38`). The existing batch key requires target language, `keep_all_outputs`, `drop_extra`, chunk T, and decoder mode to match (`src/nemotron_speech/batch_primitives.py:24-31`). The ready predicate requires both timeline and pending-audio guards to pass exactly (`src/nemotron_speech/batch_primitives.py:34-56`). The scheduler already sorts candidates by safe batch size at deadline (`src/nemotron_speech/server.py:4880-4918`), enforces lane compatibility (`src/nemotron_speech/server.py:4886-4899`), and dispatches the multi-lane batch path when available (`src/nemotron_speech/server.py:4937-4955`). The roofline still observed 86% B=1 (`proj-2026-05-23-1731/roofline-COMBINED.md:49-50`), which means "fill B" is not merely an implementation checkbox.

   Recommended change: add a trace-driven batching spike before Phase 2. Feed real per-session readiness timestamps, language, chunk T, finalize state, lane affinity, and max-wait constraints into a simulator and report achievable B distribution. Do not budget 3-5x until that trace says the workload can actually batch without adding latency.

8. [MAJOR] The single padded-T finalize bucket is counted as a cheap density win, but current graph semantics are exact-shape and static-buffer based.

   The plan says Phase 3 will use a "single padded-T_max finalize bucket" and "mask the pad" (`proj-2026-05-24-from-scratch-runtime/PLAN.md:222-228`). The current graph manager explicitly captures exact batch sizes and does not pad smaller buckets (`src/nemotron_speech/cudagraph_encoder.py:6-8`). Finalize keys include exact `batch_size`, `time_steps`, `drop_extra`, and `keep_all_outputs=True` (`src/nemotron_speech/cudagraph_encoder.py:46-54`), and the manager documents exact `(B, T, drop_extra)` finalize buckets that never pad T (`src/nemotron_speech/cudagraph_encoder.py:293-296`). Replay fails closed on exact input shape mismatch (`src/nemotron_speech/cudagraph_encoder.py:578-599`). The static length buffer is initialized to the bucket's full T (`src/nemotron_speech/cudagraph_encoder.py:193-204`), and replay copies inputs into static graph buffers before replaying (`src/nemotron_speech/cudagraph_encoder.py:272-281`).

   Padding is not just a graph-manager change. The encoder uses length and cache length to build masks and offsets (`/home/khkramer/src/parakeet/venv/lib/python3.12/site-packages/nemo/collections/asr/modules/conformer_encoder.py:645-667`), and finalize calls request `keep_all_outputs=True` (`src/nemotron_speech/server.py:7416-7428`, `src/nemotron_speech/server.py:8797-8809`). If padded frames leak into encoder outputs, decoder state, or final text, WER may stay acceptable while delta/final boundary behavior changes.

   Recommended change: move padded-T into a separate proof gate before native Phase 3, ideally in the near-term Python plan where the current graph manager exists. Require byte-exact real-frame encoder outputs, cache outputs, and final event streams for every T in the current 42..60 range before counting K=4 density.

9. [MAJOR] The native scheduler correctness plan hand-waves the hardest part: ownership of mutable per-session state across worker threads.

   The plan says Rust borrow checking or C++ TSan/ASan will enforce concurrency safety (`proj-2026-05-24-from-scratch-runtime/PLAN.md:151-157`, `proj-2026-05-24-from-scratch-runtime/PLAN.md:210-220`). The current Python code uses per-session `state_lock` (`src/nemotron_speech/server.py:486-490`), sorted lock acquisition for batch dispatch (`src/nemotron_speech/server.py:5086-5089`, `src/nemotron_speech/server.py:5190-5192`), generation numbers to suppress stale output (`src/nemotron_speech/server.py:5201-5203`, `src/nemotron_speech/server.py:5220-5225`), and explicit in-flight bookkeeping (`src/nemotron_speech/server.py:5130-5134`, `src/nemotron_speech/server.py:5272-5274`). Finalize has separate fork isolation and generation checks (`src/nemotron_speech/server.py:7621-7637`, `src/nemotron_speech/server.py:7660-7679`, `src/nemotron_speech/server.py:7774-7775`).

   TSan does not prove logical ordering: it will not tell you that a stale final was emitted after a newer reset, or that a hypothesis object was aliased across batch rows. The batch primitive explicitly guards against hypothesis aliasing because NeMo mutates hypotheses in place (`src/nemotron_speech/batch_primitives.py:109-126`).

   Recommended change: define the native state ownership model before Phase 1. Use a per-session actor or explicit immutable snapshot/result-commit protocol, and include generation/stale-output rules in the runtime API. TSan/ASAN should be secondary validation, not the design.

   Recommended change: rewrite B4 as "Python runtime/scheduler rewrite on free-threaded CPython," not "keep current Python." Spike 0.3 should require a real off-event-loop dispatcher and measure end-to-end scheduler tail, not only import/run and a synthetic launch ceiling.
