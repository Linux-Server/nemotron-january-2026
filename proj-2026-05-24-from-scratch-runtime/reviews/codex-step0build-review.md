# Codex Step-0 Build Review

Scope reviewed: `runtime/cpp/density_main.cpp`, the `runtime/cpp/CMakeLists.txt` diff, all JSONL telemetry under `runtime/artifacts_n200_mel/logs/`, `codex-jobs/step-0-task-mpmqloo6-28ary2.log`, and `PHASE2-PLAN.md` Step 0 plus the ownership/topology contract.

## BLOCKER

1. **The scalar-sync gate is currently a false STOP criterion.** `PHASE2-PLAN.md:77-80` defines `.item()`/scalar-sync p95 <=5% as a PASS condition and >10% as a STOP candidate, and the harness implements that literally in `runtime/cpp/density_main.cpp:950-953` and rolls it into the final Step-0 exit in `runtime/cpp/density_main.cpp:1593-1600`. The smoke artifact then fails 0b solely because `item_wait_pct_of_steady_gpu.p95=63.8401` with `mismatches=0` (`runtime/artifacts_n200_mel/logs/density_num_runners2_stream-explicit_topology-one_process_shared_steady_loader_per_thread_session_handles_20260526T145443Z.jsonl:1`; summarized in `codex-jobs/step-0-task-mpmqloo6-28ary2.log:261-264`). That high single-thread item wait is the overlap opportunity the multi-thread design is meant to fill, not by itself evidence of a bad topology. The real failure condition is whether `.item()` causes a global/default-stream synchronization or serializes unrelated worker streams. **Fix:** split 0b into `identity_pass` and `scalar_locality_pass`; keep `item_wait_pct` as reported opportunity/tail telemetry, and gate on an explicit local-vs-global sync test: same decode with default-stream control, an independent-stream sentinel kernel/CUDA event probe around `.item()`, and Nsight/CUPTI evidence that explicit-stream `.item()` waits do not drain unrelated streams. Do not issue a STOP from the 5% ratio alone.

2. **0a can report PASS without the required profiler proof or steady correctness proof.** Step 0a requires one shared loader, explicit streams, concurrent==serial correctness, profiler kernels on >=2 streams, and flat memory (`PHASE2-PLAN.md:72-76`). The 0a harness times `loader.run()` and only checks `out.size() >= 5` (`runtime/cpp/density_main.cpp:1067-1095`); it never compares steady outputs to a serial oracle. The pass expression only checks speedup, memory ratio, default-stream penalty, and run errors (`runtime/cpp/density_main.cpp:1276-1283`); no Nsight/CUPTI artifact or stream-kernel counter is required. The built-in `cuda_event_overlap_estimate` is not sufficient: it is just `sum(cuda_event_ms)/wall_ms` (`runtime/cpp/density_main.cpp:1171-1174`), and the default-stream control reports overlap estimates >1 despite `unique_streams=1` (`runtime/artifacts_n200_mel/logs/density_num_runners4_stream-default_topology-negative_default_stream_shared_steady_loader_20260526T145342Z.jsonl:1`). **Fix:** add a serial steady-output oracle for every steady case and compare the AOTI outputs from each worker/N run; require an Nsight Systems/CUPTI trace or machine-readable profiler summary showing kernels actually issued on >=2 non-default streams with overlap. Treat the event overlap scalar as a diagnostic only.

3. **Required topology negative controls are missing.** The topology contract requires mutex-serialized, default-stream, and `num_runners=1` controls (`PHASE2-PLAN.md:61-66`), and STOP corroboration requires a topology sweep including `num_runners=1` vs `=N` (`PHASE2-PLAN.md:153-156`). The harness always couples workers and runners in the main sweep (`runtime/cpp/density_main.cpp:1236-1243`), so it never runs `workers=N, num_runners=1`; it also has no mutex-serialized mode. For 0b, the decode/scalar path always uses explicit streams (`runtime/cpp/density_main.cpp:889-913`), so the default-stream control does not exercise the `.item()` path whose locality is under review. **Fix:** add independent `--workers` and `--num-runners`, a `--mutex-serialize-run` control, and a 0b default-stream mode. The minimum matrix before PASS/STOP is `workers=N,num_runners=N,explicit`, `workers=N,num_runners=1,explicit`, `workers=N,num_runners=N,default`, and `workers=N,num_runners=N,explicit+mutex`, with the same correctness assertions.

## MAJOR

1. **The memory-flat / one-weight-copy evidence is confounded.** The code stores the prepared steady cases as GPU tensors (`runtime/cpp/density_main.cpp:999-1030`) and starts the N sweep immediately after preparation (`runtime/cpp/density_main.cpp:1231-1238`) without a cleanup before the N=1 baseline. It then computes flatness from peak total GPU-used ratios (`runtime/cpp/density_main.cpp:1254-1266`). The artifact shows the problem: N=1 peak is 6.71GB while N=16 peak is 5.34GB, yielding `mem_ratio_target_vs_n1=0.796246` (`runtime/artifacts_n200_mel/logs/density_num_runners0_stream-explicit_topology-shared_steady_loader_runner_pool_summary_20260526T145342Z.jsonl:1`). A lower peak at N=16 than N=1 is not proof of one constants copy; it is proof the measurement is polluted by allocator/process state. **Fix:** run each N in a fresh process or call cleanup before the first measured loader, log loader delta (`used_after_loader - used_before_loader`) separately from activation/run peak, and add per-process NVML/CUDA allocator memory if available. Use target-N fresh-process deltas for the one-weight-copy gate.

2. **The steady overlap workload reuses the same GPU input tensors across workers.** `build_steady_cases()` creates one vector of GPU-resident input tensors (`runtime/cpp/density_main.cpp:999-1030`), and every worker loops over those same `SteadyCase` objects (`runtime/cpp/density_main.cpp:1144-1147`). That is weaker than production topology, where each session has distinct cache/input buffers, and it can hide input/cache bandwidth and aliasing issues. **Fix:** build or clone per-worker input/cache tensors, preferably from distinct utterance/chunk sequences, and include a shared-input negative/diagnostic only if useful.

3. **The logged evidence is smoke, not a Step-0 pass candidate.** The archived job is honest in prose that the GPU runs were limited (`codex-jobs/step-0-task-mpmqloo6-28ary2.log:252-268`), but the artifact directory contains both a 0a-only pass summary at stamp `20260526T145342Z` and a later failed smoke summary at `20260526T145443Z` (`runtime/artifacts_n200_mel/logs/density_num_runners0_stream-explicit_topology-shared_steady_loader_runner_pool_summary_20260526T145342Z.jsonl:1`, `runtime/artifacts_n200_mel/logs/density_num_runners0_stream-explicit_topology-shared_steady_loader_runner_pool_summary_20260526T145443Z.jsonl:1`). The 0b/0c artifacts are only N=2, rows=2 (`runtime/artifacts_n200_mel/logs/density_num_runners2_stream-explicit_topology-one_process_shared_steady_loader_per_thread_session_handles_20260526T145443Z.jsonl:1`). Also, skip modes can still print a final `DENSITY STEP0 PASS` because skipped gates are accepted in the final boolean (`runtime/cpp/density_main.cpp:1574-1576`, `runtime/cpp/density_main.cpp:1593-1600`). **Fix:** add `smoke=true/false`, `partial=true/false`, and a run manifest marking the canonical full run; put each run stamp in its own directory or cleanly archive stale logs. If any gate is skipped or rows<N corpus, the final status should be `PARTIAL_DIAGNOSTIC`, not `STEP0 PASS`.

4. **0c does not yet meet the full finalize gate.** The plan requires same/mixed/hot bucket, token/event/WER regression, collector fields, stale-gen-sensitive events, and finalize wait thresholds (`PHASE2-PLAN.md:81-83`). The code exercises same and mixed bucket selections (`runtime/cpp/density_main.cpp:1539-1545`), but the recorded smoke is only two workers and two rows, and there is no WER or stale-generation fixture in the 0c telemetry (`runtime/artifacts_n200_mel/logs/density_num_runners2_stream-explicit_topology-shared_finalize_bucket_runner_pool_same_bucket_20260526T145443Z.jsonl:1`, `runtime/artifacts_n200_mel/logs/density_num_runners2_stream-explicit_topology-shared_finalize_bucket_runner_pool_mixed_bucket_20260526T145443Z.jsonl:1`). **Fix:** run 0c over the full 200 rows, add a hot-bucket target-N case, and include stale-generation/event fixtures or explicitly defer that part out of Step 0 in the plan before calling 0c PASS.

## MINOR

1. **Per-worker stream uniqueness is logged for 0a but not gated or logged for 0b/0c.** The stream pool call ignores the `worker` argument and relies on pool allocation (`runtime/cpp/density_main.cpp:349-351`); 0a records `unique_streams` (`runtime/cpp/density_main.cpp:1112-1120`, `runtime/cpp/density_main.cpp:1200-1202`), but the pass expression does not assert `unique_streams == workers`, and 0b/0c do not emit stream handles/unique counts. **Fix:** assert and log unique stream count for every explicit-stream gate; fail topology if any explicit worker shares a stream unexpectedly.

2. **The CMake diff itself is mechanically fine but bare.** The target addition at `runtime/cpp/CMakeLists.txt:58-64` matches the existing link/include pattern and I do not see a build-system blocker. A small `add_custom_target(density DEPENDS density_main)` would make it symmetric with `session`, but this is convenience, not a correctness issue.

## Verdict

The harness is a useful first diagnostic and mostly wires the intended ownership topology: one shared steady AOTI loader with `num_runners=N`, explicit stream handles into `run(inputs, stream_handle)`, and per-worker session/TorchScript handles. However, the Step-0 gates are not yet sound as PASS/STOP gates. The current evidence supports only: "smoke correctness at N=2 was clean, steady throughput improves over N=1, default stream is worse in one steady control, and fuller profiler/full-corpus proof is still missing."

Before a Step-0 PASS/STOP call, fix the scalar-sync gate semantics, add the missing topology controls, require profiler proof for 0a overlap, repair the memory-flat measurement, and run full 0b/0c rather than smoke. As specified and implemented today, the harness is not yet sufficient for a definitive Step-0 PASS, and the scalar-sync threshold is incorrectly specified for STOP.
