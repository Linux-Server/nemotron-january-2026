## Context
The C++ `ws_server` warm-cache time-to-serving is already ~18s (background warmup, `6475bfe`), but on a **cold boot / fresh box** the startup is dominated by reading + extracting up to ~3×2.48 GB of duplicate encoder weights + AOTI packages from disk. This phase cuts the cold-boot load path via (1) eliminating duplicate 2.48 GB weight loads (the biggest win), (2) a conditional AOTI extraction cache + `/tmp`-leak fix, (3) safe load parallelization, (4) page-cache prewarm, and (5) rolling the latest cold-start work onto the L40S boxes. Every change must stay byte/token-exact vs the shipped runtime (enc_first is the one place we accept tolerance-equal + token/event-exact, because it can only be AOTI'd with a small numeric diff — see Rules).

## Rules

### Correctness / parity (hard gates)
- BYTE/TOKEN EXACT vs the shipped runtime for every step that touches load/bind/runtime — EXCEPT the `enc_first` AOTI path, which is **tolerance-equal (max_abs≈0.004) + token/event-EXACT**: there it must produce identical tokens/events end-to-end (the first chunk feeds everything downstream, so a numeric diff that flips ANY token fails the gate). Keep the TorchScript `enc_first` as a default-able fallback (`NEMOTRON_WS_ENC_FIRST_TS=1`) until AOTI parity is proven on the full corpus.
- **Tiered validation (per step):** FAST gate every step — `density_main artifacts --mode b2-t1 --correctness-rows 4` (0 div, `dispatcher_exceptions=0`) + `--mode runtime-smoke` (0 div). MILESTONE gate (after the unify lands, after cache/parallel, and before ANY default-on flip) — full compat oracle 8/8 `COMPAT PASS` + perf-gate + `--ws-steady-shadow STEADY_SHADOW_REPORT verdict=PASS`. For enc_first steps add an explicit FIRST-INTERIM/first-chunk token parity check, not just finalize.

### Lifetime / ownership
- Borrowed `user_managed=true` constants MUST outlive every loader that binds them. Introduce a single `SharedEncoderConstants` owner declared BEFORE finalize_loaders, scheduler, and (AOTI) enc_first; all three borrow from it. Never let the owner be destroyed before a borrower (member-declaration order = reverse destruction order).
- Lazy-initialized loaders (deferred inline `enc_steady`) use `std::once_flag`/mutex; any lazy load must be logged/instrumented so it can't hide as first-request latency.

### Build / environment
- Build in-container (`container/enter.sh`), target `cpp/build_step10`. Re-export/strip via the venv (5090: `.venv/bin/python`; sm_89 box: `torch280-sm89-venv` + the noexecstack/libcuda link fixes in `run_l40s_density.sh`). Artifacts gitignored; tooling + manifests committed; regenerated artifacts on disk (+ S3 for L40S).

### Measurement methodology (cold vs warm)
- `COLD_START_PHASE phase=… elapsed_ms=… cumulative_ms=…` is the instrument; report each phase + `sync_warm_done` + total, and `maxjit_ms` from the realtime loadgen for any serving check. **WARM** = repeated start, page cache hot. **COLD** = drop page cache (`echo 3 | sudo tee /proc/sys/vm/drop_caches`) or fresh box. Always state which. A COLD baseline is mandatory BEFORE any code (Step 0) so each lever's target is sized.

### Safety / sequencing
- `/tmp`-leak: AOTI extraction leaks 2.4 GB temp dirs on non-graceful exit (filled `/tmp` twice). Servers exit cleanly in tests; never `pkill -f` a pattern matching the launcher.
- **`torch::jit::load` GLOBAL-REGISTRY LIVELOCK:** concurrent `jit::load` calls livelock on torch's global registry. The rule is ONE global serialized `jit::load` critical section (a single process-wide mutex / thin `load_jit_serialized(...)` wrapper) that EVERY `torch::jit::load` call in the tree routes through — **specified as a MECHANICAL SWEEP, not a hand-curated list**: the implementation MUST `grep -rn 'torch::jit::load\|jit::load(' cpp/` and replace EVERY call site with the wrapper, then add a CI/grep guard that fails if a raw `torch::jit::load` reappears. (A hand-enumerated list has proven incomplete twice — do not rely on one.) **Grep-pattern audit (done 2026-05-30):** `torch::jit::load`/`jit::load(` is the complete pattern for THIS tree — 34 sites; there are NO alternative load symbols in use (no `torch::jit::import_ir_module`, no `torch::load`, no `using namespace torch::jit` aliasing). `load_module_on_device` (runtime.cpp:331, density_main.cpp:1065) is itself a thin wrapper around `torch::jit::load`, so routing those two definitions through `load_jit_serialized` covers all their callers at once. Illustrative, non-exhaustive sites the sweep WILL catch: `bundle` (runtime.cpp:737), `load_module_on_device` preproc/joint/predict (runtime.cpp:94) + TS enc_first, the hidden load in `load_shared_constants` (session.cpp:1637-1639), the hidden load in `BatchedSteadyLoaderSet`'s no-borrow ctor (steady_batch_primitive.h:113, path :222), the replay-binary loads (session.cpp:5198/5214/5217/5222/5225/5261), and the density/selftest loads (density_main.cpp:1066/1110/6103, ws_server.cpp:2507). Step 10 may ONLY overlap AOTI construction/`dlopen` and raw prewarm I/O with the (serialized) jit loads — never two jit loads in flight.
- **AOTI extraction `/tmp` is NOT env-redirectable:** torch 2.8.0 `model_package_loader.cpp create_temp_dir()` hard-codes `/tmp/XXXXXX` (no `TMPDIR` honor) and stores it in the private `temp_dir_`. So the Step-8 `/tmp`-hygiene path must be **startup cleanup of owned stale `/tmp/*/data/aotinductor*` extraction dirs** (or a custom extract-once loader), NOT a `TMPDIR` redirect.
- **enc_first AOTI parity gate = a NEW full-1000 enc_first shadow.** The existing `aoti_full1000_shadow.py` is STEADY-only (eager first-chunks, AOTI for non-first steady chunks only — `:51`); it does NOT exercise AOTI enc_first. The gate for flipping AOTI enc_first default-ON is a NEW/extended full-1000 shadow that uses AOTI for the FIRST chunk (current path for the rest) and asserts 0 token/event divergences — b2-t1's 4 rows are insufficient given the ~0.004 numeric drift.
- Probe gates: Step 1 (weights + AOTI-first-chunk parity + cache API) gates Steps 4-5 (enc_first unify) and Step 7-8 (cache). If a probe is NO-GO, document and drop/shrink the dependent step.
- Each step independently committable + green on its tier gate before commit. Env-gate risky changes default-OFF with a fallback.
