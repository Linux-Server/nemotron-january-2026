# Round 1 Codex review

## Critical (must-fix before implementation)
- The multi-box provisioning command in the planned runbook will silently reuse one EC2 instance. `PLAN.md:215-217` sets `NEMOTRON_EC2_ITYPE` and `NEMOTRON_EC2_STATE` per box, but not `NEMOTRON_EC2_NAME`. `ec2-bench/ec2_up.py:23` defaults the instance name to `nemotron-bench-` + instance type, and `ec2-bench/ec2_up.py:44-50` reuses any pending/running instance with that name. Result: `box1`, `box2`, etc. can all point at the same L40S while writing different state files. The runbook must require unique `NEMOTRON_EC2_NAME=nemotron-asr-boxN` as well as unique `NEMOTRON_EC2_STATE`, and teardown must use the matching state files (`ec2-bench/ec2_down.py:15-17`).
- The runbook's code-copy shape does not match the launcher shape. Step 1 mandates running `server.py` from `NEMOTRON_APP_DIR` (`PLAN.md:119-120`), and the inherited launcher expects a flat app dir containing `server.py` plus sibling helper modules (`deploy/launch_multiproc.sh:15`, `deploy/launch_multiproc.sh:77`). But the runbook says to `rsync src/ to $HOME/nemotron` (`PLAN.md:218`), which would put the entrypoint at `$HOME/nemotron/src/nemotron_speech/server.py`, not `$HOME/nemotron/server.py`. The existing working push path flattens the files (`ec2-bench/ec2_push.sh:14-20`, `ec2-bench/ec2_push.sh:26`). Fix the plan to choose exactly one deploy layout: flat-copy `server.py`, `batch_primitives.py`, and `cudagraph_encoder.py`, or run the package/module with a real install/PYTHONPATH.
- The systemd unit plan is likely to bake broken paths if it "mirrors the launcher" with shell-style `$HOME`. `deploy/launch_multiproc.sh:15-17` and `:34` rely on bash expanding `$HOME`; `Environment=` in a unit does not run through a shell. If `PLAN.md:133-135` becomes `Environment=NEMOTRON_APP_DIR=$HOME/nemotron` or `NEMOTRON_VENV=$HOME/nemo-venv`, `launch_single.sh` will receive literal `$HOME/...` paths and fail to `cd` or execute Python. The plan should require `%h/nemotron`, `%h/nemo-venv`, `%h/hf`, or leave these unset so the launcher expands `$HOME` under `User=ubuntu`.
- The local smoke test is specified to prove the wrong overload behavior. `PLAN.md:179-181` says the `local_lb.py` smoke should assert that `maxconn` "sheds the overflow"; `ec2-bench/local_lb.py:11-13` explicitly says this stand-in closes overflow connections while HAProxy queues them. HAProxy `server ... maxconn` is a concurrency cap, not by itself a fast 1013-style shed. This matters because queued clients may never reach `server.py` admission (`src/nemotron_speech/server.py:5044-5063`) and therefore will not get the documented 1013 close. The plan needs an explicit HAProxy queue policy (`timeout queue`, possibly `maxqueue`, and expected client failure semantics) and the smoke/docs must stop equating local close with HAProxy behavior.
- `drain.sh wait-empty` is safety-critical but the plan does not pin the HAProxy `show stat` field it must parse. `PLAN.md:166-169` only says "current-session count"; in HAProxy CSV that is `scur`, not an arbitrary positional column guessed from `show stat`. A wrong parser can either restart a backend with live streams or hang forever. Require parsing the CSV header by name (`pxname=asr_pool`, `svname=<box>`, `scur==0`) and add a fixture-style validation for `wait-empty`.
- Phase 1 has no credible access-control story. `server.py` registers an unauthenticated WS route at `/` (`src/nemotron_speech/server.py:10075-10076`), and connection validation only checks query params `language`/`model` (`src/nemotron_speech/server.py:1439-1444`). The plan's runbook only mentions `MY_IP` generically for security-group ingress (`PLAN.md:213-214`), while `ec2_up.py` opens only SSH (`ec2-bench/ec2_up.py:78-81`) and the LB setup omits client-to-LB and LB-to-backend SG rules (`PLAN.md:223-225`). Before calling this production-ready, Phase 1 must require one concrete control: private backend IPs plus SG rules that allow `:8080` only from the LB security group, and `:8443` only from an approved client CIDR/VPN/office range. App auth can defer only if the network allowlist is explicit and blocking.

## Material (should-fix; folding into the plan adds real value)
- TLS is hand-waved at the exact point where the runbook becomes load-bearing. `gen_haproxy.py` has optional `--tls-pem` (`PLAN.md:145-151`), and the runbook shows `--tls-pem ...` (`PLAN.md:223-224`), but there is no source, file format, installation path, permission, `ws://` vs `wss://`, or renewal procedure. "No TLS automation" is listed as an accepted risk (`PLAN.md:234-235`), which is fine, but Phase 1 still needs a manual cert procedure or a hard statement that plaintext is allowed only on a private network.
- The env-var documentation is primed to be wrong. `PLAN.md:43-47` drops `NEMOTRON_` prefixes for variables that are prefixed in the real launcher: `NEMOTRON_FINALIZE_SILENCE_MS`, `NEMOTRON_WARMUP_MS`, `NEMOTRON_SCHEDULER_B1`, `NEMOTRON_BATCH_SCHED`, `NEMOTRON_BATCH_MAX_SIZE`, `NEMOTRON_BATCH_MAX_WAIT_MS`, `NEMOTRON_MODEL_LANES`, `NEMOTRON_BATCH_BARRIER_DRAIN`, `NEMOTRON_BATCH_FINALIZE`, `NEMOTRON_ENCODER_CUDAGRAPH`, `NEMOTRON_ENCODER_CUDAGRAPH_MAX_B`, `NEMOTRON_ENCODER_CUDAGRAPH_FINALIZE`, `NEMOTRON_ENCODER_CUDAGRAPH_FINALIZE_PADDED`, `NEMOTRON_SYNC_COMPRESS`, and `NEMOTRON_FINALIZE_PRIORITY` (`deploy/launch_multiproc.sh:49-55`). The plan also mixes unprefixed launcher aliases (`FINALIZE_PADDED`, `SYNC_COMPRESS`, `FINALIZE_PRIORITY`, `FINALIZE_PROFILE`, `FAULTHANDLER`, `ADMISSION_MAX_BACKLOG`, `ADMISSION_MAX_READY_AGE_MS`, `FINALIZE_T_MIN`, `FINALIZE_T_MAX`; `deploy/launch_multiproc.sh:53-67`) with server-consumed `NEMOTRON_*` names (`src/nemotron_speech/server.py:900-907`). `asr.env.example` needs one canonical namespace plus documented alias/precedence rules.
- `TimeoutStopSec` is not a drain mechanism with the planned launcher behavior. `PLAN.md:131-136` implies the unit has a drain-shaped stop window, but `PLAN.md:69` and `PLAN.md:121` say SIGTERM kills the child server. That is a clean process stop, not a graceful WebSocket drain. The runbook must say only HAProxy drain plus `wait-empty` protects in-flight streams; direct `systemctl restart nemotron-asr` can cut active clients.
- The health-check semantics should be stated precisely. `/health` always returns HTTP 200 with JSON `"status": "healthy"` or `"loading"` (`src/nemotron_speech/server.py:10035-10043`). HAProxy's `option httpchk GET /health` (`PLAN.md:150-151`) will only check the HTTP response unless an explicit `http-check expect` is added. This is probably okay because `server.py` does not bind until after `load_model()` (`src/nemotron_speech/server.py:10045-10081`), but the docs should not claim HAProxy is parsing "green" JSON unless the config actually does that.
- The runbook should use private backend addresses, but the provisioner state only records the public IP. `ec2-bench/ec2_up.py:116-131` writes `"ip"` from `PublicIpAddress`; no private IP is persisted. If the LB host is in the same VPC, using public backend IPs is both slower and expands the SG problem. Add an AWS CLI snippet to fetch `PrivateIpAddress`, or update the documented state handling.
- `gen_haproxy.py` needs explicit input validation requirements. `PLAN.md:145-148` accepts IPs from CLI/file and emits them into HAProxy config. Require `ipaddress.ip_address()` validation, reject empty fleets and duplicates, and reject arbitrary server names/config fragments. Otherwise a malformed boxes file can create a syntactically valid but unsafe config.
- The validation plan does not test `drain.sh` at all. `PLAN.md:80-85` covers bash parse, shellcheck, generator assertions, and local smoke, but not runtime-socket commands, server-name matching, `show stat` parsing, timeout exit codes, or missing `socat`. Add fixture tests for `status`, `drain`, `ready`, and `wait-empty`.
- The cloud smoke is the first meaningful end-to-end check, but the plan leaves it as a placeholder. `PLAN.md:226-227` says "reference exact ec2-bench script" instead of naming one. Candidate existing client: `proj-2026-05-19-eou-endpointing/run_full1000_conc12.py`, which supports `--url`, `--concurrency`, and `--limit` (`run_full1000_conc12.py:279-285`). The runbook should name the command, a small `--limit`, expected output, and how to interpret 1013/admission counters.
- Log handling is not production-ready yet. The launcher writes child output to `server.log` (`PLAN.md:121`); the inherited pattern truncates per-process logs on restart (`deploy/launch_multiproc.sh:77-78`, `:90-93`). Decide whether logs go to journald, append with rotation, or timestamped files. Otherwise the first crash restart can destroy the evidence needed to debug it.
- `bootstrap.sh` is "proven" but not reproducible enough for production docs. NeMo is pinned (`ec2-bench/bootstrap.sh:8`, `:29-31`), but torch is explicitly unpinned (`ec2-bench/bootstrap.sh:24-27`) and `uv` is installed from a live curl script (`ec2-bench/bootstrap.sh:17-19`). If Phase 1 keeps this verbatim, `RUNBOOK-phase1.md` and `DEPLOYMENT.md` should call out the reproducibility risk and record the last-known-good versions from validation.
- Minimum observability is under-specified. Full monitoring/log shipping can defer, but Phase 1 should still document manual checks: HAProxy `show stat`/backend UP, `/health` admission attempted/admitted/rejected (`src/nemotron_speech/server.py:5017-5025`, `:10041-10042`), server restart loops, GPU memory, and what alerting is intentionally absent. Right now only open risks are listed (`PLAN.md:234-235`), not the operator's day-1 watch points.
- The `maxconn 20` narrative needs the same caveat everywhere. `PLAN.md:190-195` calls the single-proc topology SLO-robust at ~20/box but also acknowledges the sustained multi-turn assumption is unvalidated. The docs should include the p95 evidence from `deploy/DEPLOYMENT.md:23`, not only p50 (`PLAN.md:192`), and should label `--maxconn 20` as the one-utterance-per-connection default with a conservative lower value option until sustained multi-turn testing lands.

## Minor (nice-to-have; OK to defer to implementation if low cost there)
- Generated HAProxy server names `boxN` are order-dependent (`PLAN.md:148`). That is workable, but the runbook should print a stable name-to-IP table so `drain.sh drain box1` cannot accidentally target a different machine after regenerating with reordered input.
- `HAPROXY_SOCK` permissions are not documented. The generator creates `stats socket ... mode 660 level admin` (`PLAN.md:147-150`), so `drain.sh` needs either root/sudo instructions or a specific group membership for the operator.
- Add `systemd-analyze verify deploy/nemotron-asr.service` when available. `PLAN.md:80-85` covers scripts and HAProxy, but not unit-file syntax.
- The generator's `--check` mode should say how it works when output is stdout. `haproxy -c -f <out>` (`PLAN.md:152-153`) needs either `-o` output or a temporary file.
- If the complete HAProxy config is meant to be production-owned, consider documenting `user haproxy`, `group haproxy`, `log`, and `ulimit` expectations. The inherited example only has `global maxconn` and stdout logging (`deploy/haproxy.cfg.example:12-14`), which is acceptable as a design artifact but thin for an installed LB host.

## Non-findings (things I checked that look correct)
- `option httpchk GET /health` in a TCP-mode HAProxy backend is not the blocker here. The important requirements are still `server ... check` (`PLAN.md:148`), `haproxy -c` validation (`PLAN.md:77-78`, `:152-153`), and clear health semantics as noted above.
- The core server facts in the plan match the code: `/health` and `/` are registered at `src/nemotron_speech/server.py:10075-10076`; admission rejects close with code 1013 at `src/nemotron_speech/server.py:5062`; admission defaults are effectively off at `src/nemotron_speech/server.py:88-89` and `:900-915`; `/health` includes admission counters when enabled at `src/nemotron_speech/server.py:10041-10042`.
- Carrying `env -u LD_LIBRARY_PATH` into the new launcher is correct and consistent with the existing launch path (`deploy/launch_multiproc.sh:77`).
- Keeping K=3+MPS as a fallback artifact rather than reimplementing it in `launch_single.sh` matches the stated Phase 1 scope (`PLAN.md:59-66`) and the 2026-05-27 deployment note (`deploy/DEPLOYMENT.md:23`).
- Putting production artifacts under `deploy/` and this review in the `proj-2026-05-27-l40s-cluster-deploy/` directory is consistent with the repo's existing project layout (`proj-*` directories with `PLAN.md` plus review artifacts, and existing deploy artifacts in `deploy/`).
- There is no repo-root `PLAN_RULES.md`; the only one I found is under `proj-2026-05-24-from-scratch-runtime/`, so `PLAN.md:56-57` is accurate.

## Verdict
CRITICAL_FINDINGS - the plan has several silent failure paths in the hand-deploy flow (instance reuse, wrong code layout, systemd path expansion, HAProxy overflow semantics, drain parsing, and missing network access control) that should be fixed before implementation.
